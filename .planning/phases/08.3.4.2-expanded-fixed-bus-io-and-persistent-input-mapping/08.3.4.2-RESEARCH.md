**Confidence:** MEDIUM

## User Constraints (from CONTEXT.md)

### Locked Decisions
### Bus exposure target
- FLJ should target `8` total stereo input buses, including `Main`.
- FLJ should target `8` total stereo output buses, including `FLJ Main Output`.
- If the host/server path proves fewer practical user channels than the exposed ceiling, extra local strips should remain visible but disabled with a clear server-limit message.
- Phase planning and implementation should still explicitly verify what Ableton/VST3 actually expose before claiming the full target is practical.

### Local channel lifecycle
- `Add channel` should reveal the next hidden fixed slot in order rather than creating arbitrary dynamic slots.
- Removing an extra local channel should hide it and remember its prior state.
- If the removed channel is currently live, FLJ should allow removal with a short confirmation rather than blocking removal outright.
- Hidden channels are part of the fixed-slot model; removal is a visibility/state change, not destruction of the slot definition.

### Persistent input mapping
- Saved local inputs should restore to their exact previously saved fixed slot on project reopen.
- If a saved local channel was hidden when the project closed, it should stay hidden while keeping its saved mapping and state.
- Re-adding a previously hidden channel should restore its full prior state, including name, level, mute state, and exact input slot mapping.
- Input-side persistence is required for names, enabled slots, and bus-to-local-channel identity.

### Expanded remote output model
- FLJ should keep the current remote-output dropdown routing model rather than replacing it with a new assignment workflow.
- The main change is exposing more fixed output buses, up to the eight-output target, not redesigning how remote routing is selected.
- New remote users should default to `FLJ Main Output`.
- If a remote user disconnects and reconnects within the same session, their last output assignment should be preserved for that session.
- Remote output assignment should not be project-persistent across reopen; it remains session-scoped because room membership is not stable.

### Claude's Discretion
- Exact wording for disabled extra-strip messaging when the server path cannot support all exposed fixed slots.
- Exact confirmation wording and UI affordance for removing a currently live local channel.
- Exact restore-status messaging, badges, or subtle indicators when hidden channels retain saved mappings.
- Exact labels for the expanded output dropdown entries as long as they stay explicit and host-facing.

### Deferred Ideas (OUT OF SCOPE)
- Any expansion beyond the eight-input/eight-output target remains a later phase.
- Project-persistent remote output routing across reopen remains out of scope for this phase.
- Larger mixer layout, density, and visual redesign work remains Phase 9.
- Any plugin-owned enumeration of Ableton tracks or devices remains out of scope; host-owned FLJ bus routing is still the chosen model.

## Summary

The good news is that Phase 08.3.4.1 already solved the hardest protocol question. The current codebase no longer needs a second transport redesign: `CodecStreamBridge` already carries local channel metadata, `FramedSocketTransport` already writes upload-begin frames with a channel index, and the processor already publishes per-slot channel info for `Main` and `Local Send 2`. Planning for 08.3.4.2 should therefore focus on scaling the fixed-slot model, not rebuilding the codec or NINJAM transport path again.

The real work is architectural cleanup and controlled expansion. Today the processor is still hard-coded around exactly two local slots, exactly three output routes, and dedicated buffers for only `Main` plus `Local Send 2`. The editor still hard-codes one `Add channel` action that only reveals `Local Send 2`, and the route dropdown always assumes three output labels. State persistence is also split in a way that now conflicts with the locked decisions: local-strip persistence is already close to what this phase wants, but remote output assignments are currently serialized into project state even though this phase explicitly says they must remain session-scoped only.

The host-wrapper evidence is mixed in a useful way. JUCE and VST3 clearly support fixed multi-bus plugins, and Ableton officially documents up to 16 stereo outputs for multi-out plug-ins, so the eight-output target is technically conservative. The weak spot is input-side ergonomics: prior Ableton proof confirms `Local Send 2` is exposed in normal routing UI, but the official Ableton docs reviewed here do not prove an eight-input third-party effect workflow. That means the plan should implement the data model and processor loops for eight fixed slots, but keep an explicit manual Ableton validation gate before claiming the full eight-input ceiling is practical.

**Primary recommendation:** Plan Phase 08.3.4.2 as a fixed-registry expansion to `8 in / 8 out`, with loop-based slot processing, explicit persisted local-slot state keyed by stable slot IDs, session-only remote output routing, and a mandatory Ableton checkpoint that records the actual practical input/output ceiling before the phase is declared complete.

### Pattern 1: Fixed Registries For All 8 Input And Output Slots
**What:** Replace the current `2`-slot and `3`-route arrays with compile-time registries sized for the full target: `8` stereo input buses including `Main`, and `8` stereo output buses including `FLJ Main Output`.
**When to use:** Wave 0 design and Wave 1 processor changes.
**Why:** This preserves the host-owned fixed-bus model that Ableton already approved while removing the need for slot-specific branches everywhere.
**Example:**
```cpp
struct FixedLocalRoutingSlot
{
    const char* sourceId;
    int inputBusIndex;
    const char* fallbackLabel;
    bool visibleByDefault;
};

static constexpr std::array<FixedLocalRoutingSlot, 8> kFixedLocalRoutingSlots {{
    { "local-main",   0, "Main",         true  },
    { "local-send-2", 1, "Local Send 2", false },
    { "local-send-3", 2, "Local Send 3", false },
    { "local-send-4", 3, "Local Send 4", false },
    { "local-send-5", 4, "Local Send 5", false },
    { "local-send-6", 5, "Local Send 6", false },
    { "local-send-7", 6, "Local Send 7", false },
    { "local-send-8", 7, "Local Send 8", false },
}};
```
Source: JUCE `AudioProcessor` / `BusesProperties` docs, local code inspection of `src/plugin/FamaLamaJamAudioProcessor.h` and `src/plugin/FamaLamaJamAudioProcessor.cpp`

### Pattern 2: Slot Loops, Not Slot-Specific Buffers
**What:** Replace `localSend2UploadIntervalBuffer_`, `localSend2VoiceUploadBuffer_`, and the current slot-specific branches with per-slot runtime arrays or vectors indexed by fixed slot definition.
**When to use:** `processBlock()` and local upload state reset paths.
**Why:** The current implementation already has enough signal-path logic; it just does not scale beyond slot `1` without duplication.
**Example:**
```cpp
for (const auto& slot : kFixedLocalRoutingSlots)
{
    auto input = getBusBuffer(buffer, true, slot.inputBusIndex);
    auto& runtime = localSlotRuntime_[slot.inputBusIndex];

    if (! shouldPublishLocalSlot(slot.sourceId))
    {
        runtime.reset();
        continue;
    }

    submitLocalSlot(input, runtime, slot);
}
```
Source: JUCE `getBusBuffer()` docs and local code inspection of `src/plugin/FamaLamaJamAudioProcessor.cpp`

### Pattern 3: Persist Only Local Slot State, Not Remote Output Assignments
**What:** Promote local slot state to an explicit persisted model and remove remote output assignments from project serialization.
**When to use:** State schema update for this phase.
**Why:** The locked phase decisions now split persistence policy cleanly:
- local input mapping is project-persistent
- remote output routing is session-scoped only

**Example:**
```cpp
struct PersistedLocalSlotState
{
    std::string sourceId;
    std::string displayName;
    bool visible { false };
    MixerStripMixState mix;
};

// Save/restore this for local slots only.
// Do not write remote output assignments into plugin state.
```
Source: local code inspection of `src/plugin/FamaLamaJamAudioProcessor.cpp`

### Pattern 4: Remove Means Hide, With Ordered Re-Add
**What:** Treat extra local channels as permanent fixed definitions whose runtime state is hidden or shown, never created or destroyed.
**When to use:** Editor UX and processor visibility APIs.
**Why:** This exactly matches the locked lifecycle model and makes restore semantics deterministic.
**Example:**
```cpp
bool addNextLocalChannel()
{
    for (const auto& slot : kFixedLocalRoutingSlots)
    {
        if (! slot.visibleByDefault && ! getLocalSlotVisible(slot.sourceId))
            return setLocalSlotVisible(slot.sourceId, true);
    }
    return false;
}

bool removeLocalChannel(const std::string& sourceId)
{
    return setLocalSlotVisible(sourceId, false);
}
```
Source: local code inspection of `src/plugin/FamaLamaJamAudioProcessor.cpp` and `src/plugin/FamaLamaJamAudioProcessorEditor.cpp`

### Pattern 5: Derive Route Labels And Availability From The Bus Registry
**What:** Build remote-strip dropdown labels from the fixed output route registry and the current bus enablement state, instead of hard-coding three labels in `createEditor()`.
**When to use:** Editor projection and restore fallback logic.
**Why:** The current editor hard-codes exactly `FLJ Main Output`, `Remote Out 1`, and `Remote Out 2`, which will silently drift as soon as the route count changes.
**Example:**
```cpp
std::vector<std::string> labels;
for (const auto& route : kFixedRemoteOutputRoutes)
{
    auto* bus = getBus(false, route.outputBusIndex);
    if (bus != nullptr && bus->isEnabled())
        labels.push_back(route.hostLabel);
}
```
Source: JUCE `AudioProcessor::Bus::isEnabled()` docs and local code inspection of `src/plugin/FamaLamaJamAudioProcessor.cpp`

### Anti-Patterns to Avoid
- **Do not keep expanding the `Main` plus `Local Send 2` special cases.** Phase 08.3.4.2 needs loops and slot registries, not more one-off buffers.
- **Do not serialize remote output assignments into project state.** The current code does this today, but the phase decision says not to.
- **Do not key persistence from editable names or visible order.** Exact slot restore requires immutable slot IDs.
- **Do not treat removal as destruction.** Hidden channels must preserve name, mix state, and slot identity.
- **Do not promise the full 8-input Ableton path without a fresh host checkpoint.** Official docs do not prove that ceiling for a third-party effect.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dynamic bus manager | Runtime add/remove bus subsystem | Fixed JUCE `BusesProperties` registries sized for 8 | The chosen model is fixed-slot and host-owned |
| Bus-channel index math | Manual absolute channel pointer arithmetic | `getBusBuffer()` and `AudioProcessor::Bus` helpers | JUCE already owns the mapping details |
| Slot identity | Persistence keyed by display name or visible position | Stable slot IDs such as `local-send-4` | Renames and hide/show must not move state |
| Remote route persistence | Project-state restore for remote output assignment | Session-memory map only | Room membership is not stable across reopen |
| Channel removal | Destroy-and-recreate slot objects | Visibility/state toggles over fixed slot definitions | Hidden channels must restore exact prior state |

**Key insight:** The phase is mostly about making the fixed-slot model honest everywhere. The transport is already close. The missing work is array-driven processor state, explicit persistence policy, and host-validated scaling.

## Common Pitfalls

### Pitfall 1: Planning Around Eight Inputs As If Ableton Already Proved Them
**What goes wrong:** The plan assumes `8` routable extra effect inputs are a settled Live capability.
**Why it happens:** JUCE and VST3 make multi-input declaration straightforward, and the `Local Send 2` proof succeeded.
**How to avoid:** Separate "processor supports 8 fixed input slots" from "Ableton practically exposes all 8" and keep a manual ceiling checkpoint in the phase gate.
**Warning signs:** The plan marks `ROUTE-04` complete without a new Ableton matrix that exercises more than the first extra input.

### Pitfall 2: Hidden Local Slots Restore Inexactly
**What goes wrong:** A hidden channel reappears on reopen with the wrong slot, wrong name, or default mix values.
**Why it happens:** Persistence is derived from visible order or generic strip snapshots instead of stable slot identity.
**How to avoid:** Persist local slots by immutable `sourceId`, keep hidden slots in state, and restore exact slot identity before editor projection.
**Warning signs:** Re-adding a hidden slot gives the right name but the wrong bus index or default mix values.

### Pitfall 3: Remote Output Persistence Violates The Locked Scope
**What goes wrong:** Remote assignments come back after project reopen even though they are supposed to be session-only.
**Why it happens:** The current code writes `fixedRemoteOutputAssignments_` into plugin state in two places.
**How to avoid:** Remove remote route serialization from `getStateInformation()` / `setStateInformation()` and keep reconnect memory only in runtime state.
**Warning signs:** A reopened project restores a remote strip to `Remote Out 3` before that user has even rejoined the room.

### Pitfall 4: The UI Exposes Output Buses That Are Not Currently Usable
**What goes wrong:** A dropdown shows buses that the current host layout has disabled, leading to silent routes.
**Why it happens:** Editor labels are built from compile-time constants only.
**How to avoid:** Derive labels and enabled state from the route registry plus current bus enablement/layout.
**Warning signs:** The dropdown offers a bus that does not exist in the host's routing chooser.

### Pitfall 5: Removing A Live Local Channel Leaves Stale Publish State
**What goes wrong:** The strip disappears, but the server still sees stale channel info or the local mask does not update promptly.
**Why it happens:** Removal is treated as a UI-only action instead of a state and transport publication change.
**How to avoid:** On removal, hide the slot, clear or stop publishing its local channel info, and reset its upload buffers in the same state transition.
**Warning signs:** Another client still sees the removed channel after the UI has hidden it.

### Pitfall 6: Special-Casing Slot 2 Slows The Entire Phase
**What goes wrong:** Slots `3` through `8` require repeated copy-paste changes across buffers, meters, and upload branches.
**Why it happens:** The current processor still carries one-off buffers and branches for `Local Send 2`.
**How to avoid:** Refactor to arrays before expanding the count.
**Warning signs:** Every new slot requires a new dedicated member variable and another branch in `processBlock()`.

## Open Questions

1. **What is the real Ableton ceiling for extra FLJ input buses on this plugin workflow?**
   - What we know: prior host validation proved `Local Send 2` is exposed in normal Live routing UI, and JUCE/VST3 can declare many more buses.
   - What's unclear: official Live docs reviewed here do not prove an eight-input third-party effect workflow.
   - Recommendation: make a new 08.3.4.2 manual matrix that explicitly records how many of the declared input buses Live actually exposes and how discoverable they are.

2. **Should currently unvalidated extra slots be visible-but-disabled immediately, or only after the manual ceiling verdict is recorded?**
   - What we know: the locked decision says extra strips should remain visible but disabled with a clear message if the practical ceiling is lower than the exposed target.
   - What's unclear: whether the phase should ship a temporary message before the manual verdict exists.
   - Recommendation: keep the state model capable of all eight slots, but make the final disabled-slot messaging part of the manual validation closeout so the message matches the real host result.

3. **What is the safest runtime key for session-only remote reconnect memory?**
   - What we know: current remote identity already uses stable `sourceId = groupId#channelIndex` semantics for active room channels.
   - What's unclear: whether any reconnect path can change channel index for the same user within one session.
   - Recommendation: store session route memory by `sourceId`, and document the residual risk if a real-room reconnect changes channel numbering.
