## Summary

Phase `08.3.4.1` already moved FLJ from a one-bus proof to a small fixed-bus runtime: the processor currently declares `2` input buses and `3` output buses, uses `getBusBuffer()` for main/aux separation, exposes fixed local slot and remote route arrays in the processor header, and serializes visible mixer-strip state by stable source IDs. That means this phase does not need a brand-new routing architecture. It needs a disciplined expansion of the current fixed-slot model.

The safest implementation path is:
1. Replace the current hard-coded `2`-local / `3`-output limits with fixed arrays sized up to the `8`-bus target.
2. Keep the model fixed-slot, not dynamic-bus.
3. Continue using processor-owned slot descriptors and serialized per-source strip state.
4. Treat input persistence as stable-slot restoration by source ID and bus index.
5. Treat remote output assignment as session-only state layered on the existing dropdown route map.

The largest unknown is not JUCE support. JUCE and VST3 can represent multiple buses. The unknown is how many additional inputs are practically exposed and routable in Ableton for this plugin shape beyond the already-proven `Local Send 2` path. Output scaling is lower-risk because Ableton already exposed the current extra outputs and the prior research documented multi-output host workflows more clearly than multi-input effect workflows. This phase should therefore plan automated expansion work and pair it with explicit manual Ableton validation that records the real practical ceiling.

## Recommended Architecture

### Pattern 1: Fixed Ceiling Arrays, Not Dynamic Buses
Scale the existing `kFixedLocalRoutingSlots` and `kFixedRemoteOutputRoutes` arrays to the `8`-bus target rather than introducing runtime bus add/remove. This matches the current architecture, keeps source IDs stable for persistence, and avoids reopening the dynamic-host-behavior risks already rejected in the previous research phase.

### Pattern 2: Processor-Owned Slot Runtime
Keep all local-slot visibility, label, mix, and bus-index ownership in processor state. The editor should continue to reflect visible strips instead of inventing UI-local lifecycle state. This is already how the current add/hide behavior works and is the safest place to implement exact-slot restoration.

### Pattern 3: Session-Scoped Remote Route Cache
Remote output routing should remain a sourceId-to-output-bus assignment map, but reopen persistence should be trimmed or ignored for remote strips. Same-session reconnect restoration is still safe because the session owns the current remote identities; cross-reopen persistence is not.

### Pattern 4: Host-Proven Ceiling Instead of Assumed Ceiling
Code should expose the target bus count, but the validation contract must record what Ableton actually exposes and what is usable. The phase should close with an explicit verdict such as:
- `8 inputs / 8 outputs practical`
- `N inputs / 8 outputs practical`
- or `Input ceiling lower than target due to host exposure limits`

That verdict is part of the required output of the phase, not a later inference.

## Implementation Guidance

### Bus Expansion
- Expand constructor `BusesProperties()` from the current `2` inputs / `3` outputs to fixed declarations up to the `8`-bus target.
- Replace the current size check in `isBusesLayoutSupported()` with target-aware fixed counts.
- Use loops or fixed helper accessors instead of hand-written `auxOutput2`-style indexing so the code remains maintainable at the larger fixed ceiling.

### Local Channel Lifecycle
- Keep `Main` visible by default.
- Treat every additional local slot as hidden by default until revealed.
- `Add channel` should reveal the next hidden slot in source-ID order.
- Removing should toggle visibility and active send behavior, not erase saved strip state.
- Live-channel removal should use a confirmation seam in the editor but should not require new runtime transport semantics.

### Input Persistence
- Persist by stable local slot source ID, not display name.
- Restore the exact saved bus slot when reopening.
- Preserve hidden state separately from mix/name state so a hidden slot can stay hidden while keeping all prior settings.
- Keep the current state tree approach; extend it rather than replacing it.

### Remote Output Model
- Keep the current per-remote-strip dropdown.
- Default new remote sources to main output.
- Preserve same-session route assignment if the same remote source reconnects.
- Do not restore remote route assignments from saved project state after reopen unless the planner deliberately uses a session-only snapshot path that is never serialized.

## Risks And Pitfalls

### Pitfall 1: Scaling Hard-Coded Bus Access Instead of the Slot Model
Current runtime still has explicit named buffers like `auxOutput2`. Expanding to `8x8` by copying this pattern would create fragile code. Move to slot/route iteration before widening the bus count.

### Pitfall 2: Persisting Remote Outputs the Same Way as Local Inputs
The current save/restore seams already serialize mixer-strip state broadly. If the execution phase simply keeps extending that behavior, it may accidentally preserve remote output assignments across reopen, which violates the locked decision for this phase.

### Pitfall 3: Assuming Ableton Will Expose All Target Inputs
The previous research already warned that input-side host ergonomics are the real risk. This phase should not equate declared buses with usable buses.

### Pitfall 4: Confusing Hidden Slots With Disabled Server Capacity
A hidden local slot and a visible-but-disabled slot are different user states:
- hidden = user has not revealed it or removed it
- disabled = slot is visible but unavailable due to server or practical channel ceiling

The runtime and UI should keep those concepts separate.

### Pitfall 5: Losing Exact-Slot Restore During Repack
Do not re-pack restored channels into the first free slot. The user explicitly chose exact-slot restoration so templates stay predictable.
