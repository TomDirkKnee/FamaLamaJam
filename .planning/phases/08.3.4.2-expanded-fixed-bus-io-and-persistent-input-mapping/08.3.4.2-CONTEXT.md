# Phase 08.3.4.2: Expanded Fixed-Bus I/O And Persistent Input Mapping - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the current fixed-bus routing model beyond the first `Main` plus `Local Send 2` proof by targeting up to eight exposed inputs and outputs, while adding removable extra local channels and persistent input-side naming/routing.

This phase is about expanding the existing fixed-bus model and proving the practical Ableton/VST3 ceiling in the current path. It is not a Phase 9 mixer redesign, not plugin-owned host track enumeration, and not project-persistent remote-user output routing across arbitrary room membership changes.

</domain>

<decisions>
## Implementation Decisions

### Bus exposure target
- FLJ should target `8` total stereo input buses, including `Main`.
- FLJ should target `8` total stereo output buses, including `FLJ Main Output`.
- If the host/server path proves fewer practical user channels than the exposed ceiling, extra local strips should remain visible but disabled with a clear server-limit message.
- Phase planning and implementation should still explicitly verify what Ableton/VST3 actually expose before claiming the full target is practical.

### Local channel lifecycle
- `Add channel` should reveal the next hidden fixed slot in order rather than creating arbitrary dynamic slots.
- Removing an extra local channel should hide it and remember its prior state.
- If the removed channel is currently live, FLJ should allow removal with a short confirmation rather than blocking removal outright.
- Hidden channels are part of the fixed-slot model; removal is a visibility/state change, not destruction of the slot definition.

### Persistent input mapping
- Saved local inputs should restore to their exact previously saved fixed slot on project reopen.
- If a saved local channel was hidden when the project closed, it should stay hidden while keeping its saved mapping and state.
- Re-adding a previously hidden channel should restore its full prior state, including name, level, mute state, and exact input slot mapping.
- Input-side persistence is required for names, enabled slots, and bus-to-local-channel identity.

### Expanded remote output model
- FLJ should keep the current remote-output dropdown routing model rather than replacing it with a new assignment workflow.
- The main change is exposing more fixed output buses, up to the eight-output target, not redesigning how remote routing is selected.
- New remote users should default to `FLJ Main Output`.
- If a remote user disconnects and reconnects within the same session, their last output assignment should be preserved for that session.
- Remote output assignment should not be project-persistent across reopen; it remains session-scoped because room membership is not stable.

### Claude's Discretion
- Exact wording for disabled extra-strip messaging when the server path cannot support all exposed fixed slots.
- Exact confirmation wording and UI affordance for removing a currently live local channel.
- Exact restore-status messaging, badges, or subtle indicators when hidden channels retain saved mappings.
- Exact labels for the expanded output dropdown entries as long as they stay explicit and host-facing.

</decisions>

<specifics>
## Specific Ideas

- The user wants `8 in / 8 out` as the working expansion target and believes Ableton can likely expose that many buses, but still wants the phase to prove the practical ceiling explicitly.
- The existing remote-output model is already the intended UX:
  - outputs are exposed by FLJ
  - remote users default to the main output
  - the user can move them with a dropdown
  - this phase should expand that model from the current two-output exposure toward eight
- Persistent output routing is intentionally not required across project restore because remote-user presence depends on who is in the session.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `FamaLamaJamAudioProcessor` already declares fixed proof buses through `BusesProperties`, including `Local Send 2` and extra remote output buses.
- `FamaLamaJamAudioProcessorEditor` already exposes the extra-local-channel affordance through the `Add channel` UI path.
- The existing processor/editor snapshot flow already carries strip-level state such as gain, pan, mute, and naming that can anchor persistent input restoration.
- Existing integration tests in `tests/integration/plugin_host_multi_io_routing_tests.cpp` already exercise host multi-I/O routing behavior and should be extended rather than replaced.

### Established Patterns
- The current routing model is fixed-slot and host-owned, not dynamic host-track discovery.
- Remote output routing is already presented as a per-remote-strip dropdown and should remain that way.
- Processor-owned snapshot/state plumbing is the preferred seam for restoring UI-visible mixer/channel state after reopen.

### Integration Points
- Bus declaration, naming, and per-bus processing live in:
  - `src/plugin/FamaLamaJamAudioProcessor.h`
  - `src/plugin/FamaLamaJamAudioProcessor.cpp`
- Local/remote strip controls and add/remove UX live in:
  - `src/plugin/FamaLamaJamAudioProcessorEditor.h`
  - `src/plugin/FamaLamaJamAudioProcessorEditor.cpp`
- Validation and routing proof coverage live in:
  - `tests/integration/plugin_host_multi_io_routing_tests.cpp`
  - `docs/validation/phase8.3.4.1-fixed-bus-routing-matrix.md`

</code_context>

<deferred>
## Deferred Ideas

- Any expansion beyond the eight-input/eight-output target remains a later phase.
- Project-persistent remote output routing across reopen remains out of scope for this phase.
- Larger mixer layout, density, and visual redesign work remains Phase 9.
- Any plugin-owned enumeration of Ableton tracks or devices remains out of scope; host-owned FLJ bus routing is still the chosen model.

</deferred>
