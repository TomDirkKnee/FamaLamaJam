## User Constraints (from CONTEXT.md)

### Locked Decisions
## Implementation Decisions

### Private server memory
- Remember private server `host`, `username`, and `password` across sessions.
- Keep remembered private entries in the existing combined server picker, with remembered entries shown before public entries.
- Only save a private server entry after a successful connection.
- When a saved private server is recalled, autofill the saved password in masked form and reuse it on connect without requiring re-entry.

### Transmit lifecycle
- Add a transmit control to the local channel strip rather than the top transport/session row.
- After a successful `Connect`, transmit should enter an amber warmup state and then switch to active transmit automatically when healthy.
- The transmit control has three clear user-facing states:
  - `Not transmitting` (`red`)
  - `Getting ready to transmit` (`amber`)
  - `Transmitting` (`green`)
- If the user clicks while transmit is in the amber warmup state, warmup is canceled immediately and the control switches to red.
- If the user switches transmit to red, only local upload stops. Remote receive, room connection, chat, sync, local monitor, and metronome continue to work.
- After a disconnect/reconnect, transmit should re-enter amber warmup and then return to green automatically rather than restoring immediately with no warmup.

### Voice-mode compatibility guard
- Phase 08.3 does not implement full voice chat transmit or receive.
- If FLJ detects a remote channel flagged as voice mode, it should not try to play that stream as normal interval audio.
- Unsupported remote voice-mode channels should be surfaced visibly in normal UI, including the mixer/status workflow, rather than only in diagnostics.
- The user should get an explicit warning/unsupported indication instead of silent failure or glitchy attempted playback.

### Mixer and session UX polish
- Add `solo` with standard multi-solo behavior: one or more soloed strips mute unsoloed strips while any solo is active.
- Keep diagnostics collapsible and hidden by default.
- Keep BPM/BPI voting as a compact numeric field plus explicit action button, not as presets or an implicit clickable field.
- Add metronome volume as a simple knob next to the metronome checkbox rather than as a full separate mixer strip.
- Move `Arm Sync to Ableton Play` inline with `Connect` and `Disconnect`, but keep it always visible and disabled with clear reasons when unavailable.
- Make the interval timing/progress bar a fixed full-width footer beneath the mixer/footer controls so it never scrolls away.

### Session layout cleanup
- Add a collapsible `Server settings` section.
- When collapsed, show a compact summary containing the current server plus connection status.
- Wrap room-chat text within the feed width and let the feed grow vertically with scrolling.
- Remove low-value duplicated or noisy session text called out during discussion, keeping the room/session UI compact rather than verbose.

### Claude's Discretion
- Exact wording and iconography for the red/amber/green transmit states.
- Exact collapsed/expanded `Server settings` affordance styling.
- Final placement details for the compact metronome knob beside the toggle, as long as it stays lightweight and readable.
- Exact unsupported-voice-channel wording as long as the state is explicit and user-facing.

### Deferred Ideas (OUT OF SCOPE)
## Deferred Ideas

- Full voice chat mode support, including chunked transmit and real-time receive/playback semantics, is a later phase.
- The larger JamTaba-inspired layout overhaul, including fully vertical mixer redesign and deeper custom control styling, remains Phase 9 work.
- More ambitious custom JUCE controls such as deeply embedded fader-in-meter visuals should be treated as later layout/parity work unless they fall out naturally during the Phase 9 redesign.

## Summary

Phase 08.3 should be planned as a narrow extension of the current architecture, not as a redesign. The codebase already follows the right seams for this: session and connection state are processor-owned, editor behavior is driven by snapshot getters and command handlers, plugin persistence already uses a wrapped `ValueTree` schema, and mixer strips are already keyed by stable `sourceId`s with persisted per-strip mix state. The phase should keep all new behavior inside those seams.

The main planning risk is not implementation complexity in isolation; it is cross-cutting state coupling. Private-server recall touches both the session serializer and the remembered-server history subtree. Transmit UX touches timing, outbound upload gating, and local-strip presentation. Voice-mode guard touches transport metadata, decode/mix suppression, and user-facing strip/status copy. UI polish touches layout, but the Phase 08.2 CPU work means the plan should avoid reintroducing unconditional rebuilds or editor-local business logic.

## Architecture Patterns

### Recommended Project Structure
```text

- app/session/      # discovery models, validation, and lifecycle helpers
- infra/state/      # session serializer and wrapped plugin state helpers
- net/              # framed transport metadata and NINJAM socket handling
- plugin/           # processor-owned runtime state and JUCE editor layout
```

### Pattern 1: Processor-Owned UI State, Editor as Projection
**What:** Keep transmit state, solo state, unsupported-voice state, and persisted recall state in the processor or model layer, then expose them through editor getters and command handlers.
**When to use:** For any behavior that must survive reconnects, state restore, or non-UI test execution.
**Example:**
```cpp
auto transportUiGetter = [this]() {
    const auto state = getTransportUiState();
    return FamaLamaJamAudioProcessorEditor::TransportUiState {
        .connected = state.connected,
        .hasServerTiming = state.hasServerTiming,
        .syncHealth = static_cast<FamaLamaJamAudioProcessorEditor::SyncHealth>(state.syncHealth),
        .metronomeAvailable = state.metronomeAvailable,
    };
};
```
Source: `src/plugin/FamaLamaJamAudioProcessor.cpp`

**Planning implication:** Add new transmit / solo / unsupported-voice fields to processor snapshots and editor strip state rather than inventing editor-only members.

### Pattern 2: Evolve the Wrapped Plugin State Schema In Place
**What:** The plugin already serializes active session settings plus extra wrapped state (`rememberedServerHistory`, metronome, mixer state) under a versioned root tree.
**When to use:** For remembered private-entry metadata, new per-strip persisted flags, and optional UI-collapse persistence.
**Example:**
```cpp
juce::ValueTree stateTree(kPluginStateType);
stateTree.setProperty(kPluginStateSchemaVersion, kPluginStateSchema, nullptr);
stateTree.addChild(settingsTree.createCopy(), -1, nullptr);
stateTree.addChild(rememberedServerHistoryTree, -1, nullptr);
stateTree.addChild(mixerStateTree, -1, nullptr);
```
Source: `src/plugin/FamaLamaJamAudioProcessor.cpp`

**Planning implication:** Prefer adding fields to the remembered-server child nodes and mixer-strip child nodes over inventing a second persistence mechanism.

### Pattern 3: Source-Keyed Strip Runtime With Sorted Snapshot Export
**What:** Mixer strips already live in a `sourceId -> runtime` map, with exported snapshots sorted by strip kind, group, channel, and source ID.
**When to use:** For adding `solo`, transmit affordance on the local strip, unsupported-voice badges, and extra strip subtitle/status state.
**Example:**
```cpp
if (lhs.descriptor.kind != rhs.descriptor.kind)
    return lhs.descriptor.kind == MixerStripKind::LocalMonitor;
if (lhs.descriptor.groupId != rhs.descriptor.groupId)
    return lhs.descriptor.groupId < rhs.descriptor.groupId;
if (lhs.descriptor.channelIndex != rhs.descriptor.channelIndex)
    return lhs.descriptor.channelIndex < rhs.descriptor.channelIndex;
return lhs.descriptor.sourceId < rhs.descriptor.sourceId;
```
Source: `src/plugin/FamaLamaJamAudioProcessor.cpp`

**Planning implication:** Keep local transmit on the existing local strip by extending strip state, not by introducing a duplicate transport-area control.

### Pattern 4: Timing-Boundary State Changes for Transmit Readiness
**What:** The processor already tracks `transmitWarmupIntervalsRemaining_` and decrements it only on interval boundaries.
**When to use:** For any user-facing transmit warmup state.
**Example:**
```cpp
if (transmitWarmupIntervalsRemaining_ > 0)
{
    --transmitWarmupIntervalsRemaining_;
    localUploadIntervalBuffer_.setSize(0, 0);
    localUploadIntervalWritePosition_ = 0;
}
```
Source: `src/plugin/FamaLamaJamAudioProcessor.cpp`

**Planning implication:** Replace the implicit "counter means warmup" model with an explicit transmit enum such as `Off`, `Warming`, `Active`, but keep boundary-driven countdown semantics in the processor.

### Pattern 5: Gate Unsupported Voice Channels Before Decode/Mix
**What:** Remote source activity already carries `channelFlags` through `RemoteSourceInfo`, and inbound frame handling already syncs strip metadata before decode submission.
**When to use:** For the voice-mode compatibility guard.
**Example:**
```cpp
syncRemoteMixerStrip(inboundFrame.sourceInfo);
remoteReceiveDiagnosticsBySource_[sourceId].lastEncodedBytes = inboundFrame.payload.getSize();
codecStreamBridge_.submitInboundEncoded(inboundFrame.sourceId,
                                        inboundFrame.payload.getData(),
                                        inboundFrame.payload.getSize());
```
Source: `src/plugin/FamaLamaJamAudioProcessor.cpp`

**Planning implication:** Insert the voice-mode check in this path before `submitInboundEncoded()`, and preserve a visible strip/status artifact so the user sees why playback is absent.

### Anti-Patterns to Avoid
- **Editor-owned transmit lifecycle:** Do not model warmup or reconnect behavior with editor timers or button text alone.
- **Masked password as real password:** Do not write placeholder mask text back into `SessionSettings.password`.
- **Solo implemented by mutating persisted mute state:** Keep `solo` separate so restore and UI semantics remain stable.
- **Voice-mode detection only in diagnostics:** A diagnostics-only note still looks like broken playback to users.
- **Phase 9 bleed:** Do not replace the current single-page layout architecture just to satisfy this release polish phase.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Private credential persistence | Separate settings file or ad hoc secure store for this phase | Existing plugin wrapped `ValueTree` state plus session serializer | The plugin already persists comparable credential data and supports restore/migration patterns. |
| Remembered entry ordering / de-dupe | New custom picker ordering code | `ServerDiscoveryModel` endpoint normalization and history rules | Current model already normalizes host keys, de-dupes, and keeps most-recent entries first. |
| Strip audibility under solo | Permanent mute rewriting across all strips | Derived `isAudible` check during mix/render | Multi-solo is a playback policy, not a persisted mute mutation. |
| Room chat wrapping | Fixed-height one-line labels with manual truncation | `juce::TextLayout`-backed row component or another width-aware text component | Current `24px` single-line labels do not satisfy wrapped scrolling chat. |
| Voice-mode guard | Trying to decode/play unsupported voice frames as interval audio | Early metadata gate + explicit unsupported strip state | Prevents glitchy playback and matches the explicit scope boundary. |

**Key insight:** This phase is mostly about adding state in the right place, not about inventing new systems. The project already has the correct persistence, transport, and editor seams.

## Common Pitfalls

### Pitfall 1: Endpoint-Only Remembered History Cannot Satisfy Private Recall
**What goes wrong:** The current remembered-server history only stores `host` and `port`, so recalling a remembered private server cannot restore username/password or differentiate multiple private identities on one endpoint.
**Why it happens:** `RememberedServerEntry` is currently endpoint-only, and the wrapped plugin state stores only host/port child properties.
**How to avoid:** Introduce a richer remembered-private entry schema in the wrapped state. At minimum store endpoint, username, password, and enough metadata to render "masked but reusable" recall behavior.
**Warning signs:** Picker recall updates host/port but leaves stale username/password, or multiple private identities overwrite each other.

### Pitfall 2: Masked Password UX Accidentally Overwrites the Real Password
**What goes wrong:** The UI shows a masked password, but the editor writes the literal mask string back into the active settings when the user connects.
**Why it happens:** `makeDraftFromUi()` currently copies `passwordEditor_` text directly into `SessionSettings.password`.
**How to avoid:** Track recalled-password provenance separately from displayed text. Only replace the stored password when the user actually edits the password field.
**Warning signs:** Successful recall reconnect works once, then later restores asterisks or placeholder text as the saved password.

### Pitfall 3: Transmit-Off Accidentally Breaks Session Connectivity
**What goes wrong:** Turning transmit red stops outbound upload, but also clears timing, room interaction, or remote receive.
**Why it happens:** The current upload and session plumbing live in the same `processBlock()` and connection lifecycle area.
**How to avoid:** Gate only the local input submission / outbound enqueue path. Leave connection lifecycle, inbound frames, room events, sync, monitor, and metronome untouched.
**Warning signs:** Red transmit causes reconnect status, empty room feed, metronome disablement, or remote strip disappearance.

### Pitfall 4: Solo Implemented as a Destructive Rewrite of `muted`
**What goes wrong:** Soloing strips rewrites the underlying mute state, so un-solo does not restore the user's prior mute choices.
**Why it happens:** The current mix state only persists `muted`, and a naive implementation may try to reuse it.
**How to avoid:** Add separate solo state and compute effective audibility as `!muted && (no solo active || strip.soloed)`.
**Warning signs:** Un-solo changes persisted mute values or state restore no longer matches the user's explicit mute choices.

### Pitfall 5: Voice Guard Added Too Late in the Pipeline
**What goes wrong:** Unsupported voice-mode streams still decode and queue, then simply fail to render or only show in diagnostics.
**Why it happens:** The easiest insertion point is often the mixer render loop, but that is already too late.
**How to avoid:** Gate at inbound-frame handling before decode submission, and keep per-source unsupported state in the strip snapshot/UI model.
**Warning signs:** CPU work still appears for flagged channels, diagnostics grow, and the user sees a strip but hears glitchy or partial audio.

### Pitfall 6: Layout Polish Reintroduces Phase 08.2 CPU Churn
**What goes wrong:** Collapsible sections, wrapped room rows, or footer moves reintroduce constant rebuilds and `resized()` churn.
**Why it happens:** The editor refresh loop already runs at `20 Hz`, and current room feed rebuilds trigger `resized()`.
**How to avoid:** Keep dirty-checking. Rebuild wrapped room widgets only when feed content or width actually changes, and keep collapse toggles as explicit events instead of timer-driven relayout.
**Warning signs:** `roomFeedRebuildCalls`, `resizedCalls`, or mixer widget rebuild counts rise again with idle UI.

### Pitfall 7: Voice-Mode Validation Is Blocked by Harness Blindness
**What goes wrong:** The implementation lands, but automated tests cannot express a remote peer with voice flags.
**Why it happens:** `MiniNinjamServer::RemotePeer` currently has no `channelFlags`, and `sendUserInfo()` always writes zero flags.
**How to avoid:** Extend the harness first or in the same first wave so voice guard logic is testable before UI polish hides defects.
**Warning signs:** Planning relies on manual-only verification for the most failure-prone protocol edge in the phase.

## Open Questions

1. **Should collapse state persist across plugin restore?**
   - What we know: The phase requires collapsible diagnostics and server settings, but persistence was not explicitly requested.
   - What's unclear: Whether users expect their collapsed/expanded preference to survive reopen.
   - Recommendation: Treat persistence as discretionary. If cheap, serialize it in the wrapped plugin state; otherwise keep it session-only and document that as non-blocking.

2. **Should local monitor participate in solo?**
   - What we know: The requirement says standard multi-solo behavior and the local strip is part of the mixer.
   - What's unclear: Whether the user wants local monitor always audible regardless of solo.
   - Recommendation: Default to standard behavior, including local monitor in solo logic, unless UX review explicitly asks for "always hear self."

3. **What is the authoritative voice-mode predicate?**
   - What we know: The current diagnostics treat `channelFlags == 2u` as `voice`, and `channelFlags` already reaches `RemoteSourceInfo`.
   - What's unclear: I did not independently verify the protocol-level constant from official NINJAM documentation.
   - Recommendation: Centralize the predicate in one helper and extend the integration harness to inject arbitrary flags so the implementation is locally verifiable even if the exact constant later needs correction.
