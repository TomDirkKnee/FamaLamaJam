## Summary

The current processor-side discovery merge is still the Phase 08.1/08.3 shape: `getServerDiscoveryUiState()` emits remembered rows first, deduplicates by endpoint, and drops any public Ninbot row whose `host:port` already exists in remembered history. That is why a successful connect to a public server currently replaces the richer Ninbot row with a plain remembered row. The persistence path itself is already correct for this phase: remembered credentials are only updated from `rememberSuccessfulServer()` after a real `Connected` event transitions the lifecycle to `Active`, and failed connections do not write to the store.

The smallest safe implementation is to change the merge layer, not the store or editor password-mask flow. Keep remembered history in `rememberedServers_` exactly as today, but build discovery UI rows by endpoint-key overlay: public rows first, with remembered `username/password` injected when a public endpoint matches remembered history, then append unmatched remembered entries as private-only rows. That preserves Ninbot room/user metadata, keeps successful credential recall working through the existing editor selection path, and avoids introducing any new persistence or auth concepts.

The UI polish should stay in the current single-page editor. The existing editor already has the right seams: a collapsed summary label, a separate timing label, a sync-assist button plus helper label, a right sidebar with vote controls/chat feed, a diagnostics toggle/editor, and test hooks for summary text, discovery selection, diagnostics state, and room controls. This phase should tighten those seams rather than redesigning layout ownership.

**Primary recommendation:** Implement an endpoint-key credential overlay in `FamaLamaJamAudioProcessor::getServerDiscoveryUiState()`, keep successful-connect persistence unchanged, and make editor-only copy/layout updates for the compact summary, shared vote row, diagnostics takeover, and conditional chat autoscroll.

### Pattern 1: Endpoint-Key Overlay Merge In The Processor
**What:** Build discovery UI rows from public data first, keyed by `makeDiscoveryEndpointKey(host, port)`, and overlay remembered `username/password` when a remembered entry exists for that endpoint. After all public rows are emitted, append remembered entries whose endpoints were not present in the Ninbot list.

**When to use:** Only in `FamaLamaJamAudioProcessor::getServerDiscoveryUiState()`. Do not change `cachedPublicServers_`, `rememberedServers_`, or the store format for this phase.

**Why:** This is the smallest place that can satisfy all of the server-recall requirements without touching connection/auth flow.

**Example:**
```cpp
// Source: src/plugin/FamaLamaJamAudioProcessor.cpp
std::unordered_map<std::string, app::session::RememberedServerEntry> rememberedByEndpoint;
for (const auto& remembered : rememberedServers_)
    rememberedByEndpoint.emplace(makeDiscoveryEndpointKey(remembered.host, remembered.port), remembered);

for (const auto& publicEntry : cachedPublicServers_)
{
    const auto key = makeDiscoveryEndpointKey(publicEntry.host, publicEntry.port);
    const auto it = rememberedByEndpoint.find(key);

    state.combinedEntries.push_back(ServerDiscoveryEntry{
        .source = ServerDiscoveryEntry::Source::Public,
        .label = makePublicDiscoveryEntryLabel(publicEntry),
        .host = publicEntry.host,
        .port = publicEntry.port,
        .username = it != rememberedByEndpoint.end() ? it->second.username : std::string{},
        .password = it != rememberedByEndpoint.end() ? it->second.password : std::string{},
        .connectedUsers = publicEntry.connectedUsers,
        .maxUsers = publicEntry.maxUsers,
        .stale = publicServerListStale_,
    });
    seenEndpoints.insert(key);
}

for (const auto& remembered : rememberedServers_)
    if (!seenEndpoints.contains(makeDiscoveryEndpointKey(remembered.host, remembered.port)))
        appendRememberedTailRow(remembered);
```

### Pattern 2: Keep Selection-Time Credential Hydration Editor-Only
**What:** Continue to use `applyServerDiscoveryEntrySelection()` and the existing masked-password flow (`applyRememberedPassword()`, `recalledPassword_`, `showingRememberedPasswordMask_`) as the only place where selected discovery rows hydrate hidden credentials into the draft UI.

**When to use:** For both remembered-only rows and public rows carrying overlay credentials.

**Why:** The editor already knows how to preserve a remembered password through `Connect` while allowing the user to replace it by editing the password field.

**Example:**
```cpp
// Source: src/plugin/FamaLamaJamAudioProcessorEditor.cpp
void applyServerDiscoveryEntrySelection(const ServerDiscoveryEntry& entry)
{
    hostEditor_.setText(entry.host, juce::dontSendNotification);
    portEditor_.setText(juce::String(static_cast<int>(entry.port)), juce::dontSendNotification);
    usernameEditor_.setText(entry.username, juce::dontSendNotification);

    if (!entry.password.empty())
        applyRememberedPassword(entry.password);
    else
        clearRememberedPassword();
}
```

### Pattern 3: Compact Summary And Compact Timing Are Separate Lines
**What:** Use the top summary line for connection identity and target endpoint, and use the next line purely for room timing in `BPM | BPI` form. Leave lifecycle/error detail in `statusLabel_` and `authStatusLabel_`.

**When to use:** `titleLabel_`, `serverSettingsSummaryLabel_`, and `transportLabel_` copy/layout updates in the editor.

**Why:** The current editor already has separate labels; this phase is a wording/role reassignment, not a new state model.

### Pattern 4: Sidebar Mode Switch Instead Of Sidebar Stacking
**What:** In `resized()`, branch the right sidebar into two mutually exclusive modes:
- Normal mode: `Room Chat`, shared vote row, room feed, message composer.
- Diagnostics mode: diagnostics editor fills the sidebar height; room widgets are hidden.

**When to use:** `diagnosticsExpanded_` layout and visibility changes.

**Why:** The current code stacks diagnostics under chat in a fixed 180 px slice, which is exactly what this phase wants to remove.

### Pattern 5: Shared Vote Dispatch With XOR Field Validation
**What:** Replace the two editor vote buttons with one shared `Vote` button that inspects both value fields:
- BPM filled, BPI empty: submit BPM.
- BPI filled, BPM empty: submit BPI.
- Both empty or both filled: reject locally with inline feedback.

**When to use:** Editor-only vote UI. The processor `submitRoomVote(RoomVoteKind, int)` API stays unchanged.

**Why:** This phase is compactness/polish, not room-protocol redesign.

### Pattern 6: Conditional Chat Autoscroll Based On Near-Bottom Snapshot
**What:** Before rebuilding the room feed, capture whether the viewport is already near the bottom; after rebuild/resized, only scroll to the end when that condition was true.

**When to use:** `refreshRoomUi()` / `rebuildRoomFeedWidgets()` around feed changes.

**Why:** There is currently no explicit autoscroll policy, so adding one at the feed-refresh seam is the lowest-risk implementation.

### Anti-Patterns to Avoid
- **Do not overwrite remembered credentials on selection:** row selection should only hydrate the draft UI, never write to `rememberedServers_` or disk.
- **Do not update remembered history on failed auth/connect:** the current post-`Connected` persistence rule is already correct and should stay the only write path.
- **Do not store a second copy of overlay credentials inside `cachedPublicServers_`:** keep fetch/cache data public-only and synthesize overlay rows at UI-state assembly time.
- **Do not turn this into Phase 9 layout work:** no new tabs, no mixer restructuring, no new side panels.
- **Do not always force room feed scroll-to-bottom:** that will break manual backlog reading and violate the phase decision.

## Common Pitfalls

### Pitfall 1: Fixing The Wrong Layer For Discovery Overlap
**What goes wrong:** credentials get restored, but public Ninbot metadata still disappears because the code still emits remembered rows first.
**Why it happens:** the current duplication policy lives in `getServerDiscoveryUiState()`, not in the store or parser.
**How to avoid:** change row synthesis only; keep store/parser behavior unchanged.
**Warning signs:** tests still show `public.example.org:2051` collapsing to a plain remembered label after a successful connect.

### Pitfall 2: Regressing Successful-Only Credential Persistence
**What goes wrong:** failed attempts overwrite the last good password for a remembered endpoint.
**Why it happens:** a developer adds persistence on `Connect` press or on discovery selection instead of leaving it on the `Connected` event path.
**How to avoid:** keep `rememberSuccessfulServer()` reachable only from `handleConnectionEvent()` when the lifecycle reaches `Active`.
**Warning signs:** a failed auth test changes `rememberedServers_` or the AppData XML.

### Pitfall 3: Compact Top Bar Accidentally Drops Actionable Status
**What goes wrong:** the top becomes concise, but reconnect/auth/timing loss guidance disappears.
**Why it happens:** existing verbose transport copy is removed without preserving `statusLabel_` and `authStatusLabel_`.
**How to avoid:** use the top two lines only for summary and timing; keep lifecycle and auth detail in their current labels.
**Warning signs:** timing-loss or auth-failure tests lose explicit text entirely.

### Pitfall 4: Shared Vote UI Breaks Existing Processor Semantics
**What goes wrong:** editor submits the wrong vote kind or allows both fields to compete.
**Why it happens:** the shared button is introduced without an XOR validation rule.
**How to avoid:** make the editor choose exactly one populated field before calling the existing `roomVoteHandler_`.
**Warning signs:** BPM and BPI statuses both go pending from one click, or the wrong processor command is observed.

### Pitfall 5: Diagnostics Takeover Leaves Hidden Widgets Participating In Layout
**What goes wrong:** diagnostics fills most of the sidebar but chat widgets still reserve space or remain visible.
**Why it happens:** current layout stacks everything unconditionally and only toggles diagnostics visibility.
**How to avoid:** gate room widget visibility/layout on `diagnosticsExpanded_`.
**Warning signs:** room feed still shrinks to a tiny viewport when diagnostics is expanded.

### Pitfall 6: Chat Autoscroll Fights The User
**What goes wrong:** reading older chat becomes impossible because every refresh forces the viewport to the bottom.
**Why it happens:** no pre-refresh near-bottom snapshot is taken.
**How to avoid:** compute distance from bottom before feed rebuild and only snap if the user was already within a small threshold.
**Warning signs:** manual scroll-up tests or manual QA immediately jump back to the newest message.

## Open Questions

1. **Where should unmatched remembered/private rows appear relative to public Ninbot rows?**
   - What we know: the phase requires public Ninbot rows to remain primary/visible.
   - What's unclear: whether unmatched remembered rows should still lead the list or trail it.
   - Recommendation: put public rows first and append unmatched remembered/private rows after them; this best matches "public rows remain primary" and is the most coherent planner target.

2. **How much vertical space should the empty compact timing line reserve when disconnected?**
   - What we know: the line should be empty when disconnected and there is no room timing.
   - What's unclear: whether to keep the line height stable or collapse spacing.
   - Recommendation: keep the label component but allow empty text; this avoids layout churn and keeps the change low-risk.

3. **What is the safest near-bottom threshold for conditional autoscroll?**
   - What we know: exact threshold is discretionary.
   - What's unclear: the best pixel threshold for JUCE viewport behavior on this layout.
   - Recommendation: start with 24 px or less than one line height and keep it editor-local so it can be tuned without touching processor logic.
