# Phase 08.3.1: Private Server Recall + Compact Session UI Polish - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Polish the current single-page JUCE session UI and server-recall behavior without starting the broader Phase 9 mixer/layout redesign. This phase covers how public Ninbot rows and remembered credentials should coexist, how the compact top/session area should read, and how the room sidebar should become denser and easier to use.

This phase does not include the larger mixer redesign. Mixer compactness, integrated meter/fader work, and pan-control redesign are explicitly deferred to the next stage.

</domain>

<decisions>
## Implementation Decisions

### Server memory and public-list behavior
- Public Ninbot rows should remain visible as the primary server rows because they carry richer room/user-count information.
- Previously used public-list servers should still be remembered, but the richer Ninbot row should remain visible instead of being replaced by a plain remembered row.
- When a Ninbot row matches a remembered `host:port`, selecting that public row should restore the remembered `username` and `password` behind the scenes.
- Remembered credentials should update to the most recent successful credentials for a matching `host:port`.
- Failed connection attempts must not overwrite the last known successful credentials.

### Top bar and compact session copy
- The old verbose top title should be replaced by a compact connected summary.
- When connected, the compact summary should read as `Connected as USER | HOST:PORT`.
- When disconnected, the compact summary should read as `Disconnected from HOST:PORT` when a target server is already filled in.
- Room timing should be shown on a second compact line, not crowded onto the same line as the connection summary.
- The compact timing line should use the readable format `100 BPM | 16 BPI`.
- When disconnected and there is no room timing, the second compact timing line should be empty.
- The sync-assist button should carry its own guidance text instead of relying on a separate descriptive label.
- When the DAW tempo is wrong, the disabled sync-assist button should read `Set DAW tempo to X`.

### Room sidebar compactness
- The room heading should become `Room Chat`.
- BPM/BPI voting should use one shared `Vote` button with separate BPM and BPI value fields beside it.
- Only one of BPM or BPI may be filled at a time before pressing the shared `Vote` button.
- The existing BPM/BPI range helper text should be removed to save space.
- The diagnostics toggle should move to the top area inline with the server settings controls.
- When diagnostics is expanded, the diagnostics content should take over the chat sidebar and fill the full available vertical space there.
- Chat text should wrap cleanly within the room-feed width.
- The chat feed should auto-scroll only if the user was already near the bottom.

### Claude's Discretion
- Exact wording and spacing of the compact top bar, as long as it stays concise and matches the decisions above.
- Exact inline layout of the BPM field, BPI field, and shared `Vote` button.
- Exact threshold for deciding whether the chat feed is "near the bottom" for auto-scroll behavior.
- Exact placement of the diagnostics toggle in the top area, as long as it stays clearly associated with session controls rather than the mixer.

</decisions>

<specifics>
## Specific Ideas

- The public Ninbot list is preferable for public servers because it shows live room-user information; losing that detail after connecting is the main problem this phase should correct.
- The top area should feel much more compact and informative, not like a stack of duplicated status banners.
- Diagnostics should be hidden by default, but when explicitly opened they should use the full chat/sidebar area rather than appearing as a cramped extra box.
- The user specifically wants mixer-component changes pushed to the next stage instead of letting this polish phase expand into a redesign.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/infra/net/PublicServerDiscoveryClient.cpp` already pulls the structured Ninbot public server list from `http://ninbot.com/app/servers.php`.
- `src/infra/state/RememberedServerStore.cpp` already persists remembered servers in `%AppData%\\FamaLamaJam\\remembered-private-servers.xml`.
- `src/plugin/FamaLamaJamAudioProcessor.cpp` already builds combined public/remembered discovery rows and already has the hook where remembered successful servers are merged and labeled.
- `src/plugin/FamaLamaJamAudioProcessorEditor.cpp` already contains the collapsible server-settings area, compact summary label, room sidebar, diagnostics toggle, and BPM/BPI vote controls that this phase needs to refine.

### Established Patterns
- Earlier phases already chose to keep the current single-page JUCE editor and defer the bigger layout redesign.
- The project already treats the Ninbot list as the preferred public-discovery source and uses remembered state as an assistive overlay.
- Diagnostics are already treated as optional/collapsible, which fits the decision to make them take over the room area only when expanded.

### Integration Points
- Server-memory behavior will likely center on the combined-discovery merge logic and remembered-server update path in `FamaLamaJamAudioProcessor`.
- Compact top-bar changes live in the editor's server-settings summary, transport text, and sync-assist label/button path.
- Room-sidebar changes live in the existing room feed, vote controls, diagnostics toggle, and room viewport layout code in `FamaLamaJamAudioProcessorEditor.cpp`.

</code_context>
