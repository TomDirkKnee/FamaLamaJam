## User Constraints (from CONTEXT.md)

### Locked Decisions
- Save both local and remote stems.
- Stem files must begin on interval boundaries for easy DAW import.
- If recording is enabled mid-session, files start on the next interval boundary.
- Do not pad leading silence before the recording start boundary.
- If a remote user joins later, their file starts at their first actual audio with no silence padded before it.
- Local interval stems should contain the actual transmitted interval contribution, not raw live monitor input.
- Local voice stems should also begin on the next boundary, then record continuous voice chunks from there.
- Export WAV only in this phase.
- Export dry / pre-mix stems.
- Keep the real source channel format instead of forcing stereo or mono.
- Remote interval and voice material should be written as separate files.
- Append mode in the filename only for voice files.
- Create one new session folder per recording run, named from BPM/BPI plus timestamp.
- Stem filenames should use username plus channel name, with numeric suffixes for collisions.
- Use one global `Record stems` control plus a folder picker.
- If no folder is chosen, block recording and show an inline error.
- If recording stays armed across disconnect/reconnect, keep the same session folder and restart any new files on the next interval boundary after reconnect.
- If recording is turned off mid-session, stop and close files on the next interval boundary.

### Deferred Ideas (OUT OF SCOPE)
- Per-source record toggles.
- FLAC or alternate export formats.
- Post-mix or dual dry/post exports.
- Silence backfill or metadata offsets for late joiners.
- Host multi-bus routing work (Phase 08.3.4).

## Summary

The codebase already has the right broad seam for this phase: `processBlock()` owns authoritative interval timing and sees all four audio categories that matter for export. Local interval audio is assembled into `localUploadIntervalBuffer_` before `codecStreamBridge_.submitInput(...)`; local voice audio is assembled into `localVoiceUploadBuffer_` in chunked mono form before the same bridge call; remote inbound payloads are decoded on the `CodecStreamBridge` worker thread and then re-enter the processor as either boundary-quantized interval buffers (`queueDecodedRemoteInterval(...)`) or chunked voice buffers (`queueDecodedRemoteVoiceChunk(...)`). That means stem capture does not need to tap host output or post-strip mixer state. It should tap the already-normalized dry audio objects at those existing handoff points.

The key implementation constraint is realtime safety. This phase must not do file creation, path IO, XML mutation, or WAV writes on the audio thread. The correct architecture is a new background writer service, analogous in spirit to `CodecStreamBridge`: the processor packages dry audio snapshots plus metadata (source id, user/channel labels, voice vs interval, session boundary index, export path info) into bounded work items, and a dedicated worker thread owns file/session creation and streaming WAV writes. The audio thread should only enqueue snapshots when recording is armed and a boundary-aligned start has been reached.

The other important constraint is timeline semantics. Because the user explicitly does not want leading silence, exported files should not pretend to start at a universal zero point. Instead, each file’s first sample should correspond to the first eligible interval boundary after recording armed for that source/path. That keeps files DAW-usable while staying faithful to the user’s “no silence backfill” decision. Planning should therefore treat “boundary aligned” and “common zero origin” as separate things. We need the first, not the second.

The safest plan shape is:
- Wave 1: add the recording domain/service seam plus focused RED coverage for boundary start/stop and naming lifecycle.
- Wave 2: wire local/remote taps and the global UI workflow into the current processor/editor/state flow.
- Wave 3: run a focused automated gate, rebuild the VST3, and manually validate exported stems in Ableton by importing them and checking alignment/usability.

## Existing Codebase Signals

### Local interval tap already exists at the right abstraction
`FamaLamaJamAudioProcessor::processBlock()` fills `localUploadIntervalBuffer_` against `authoritativeTiming_.samplesIntoInterval`, then submits the completed interval buffer to `codecStreamBridge_` exactly when the interval wraps. That is the correct dry, pre-mix, boundary-accurate local interval stem source.

### Local voice tap already exists before codec submission
The same processor builds mono `localVoiceUploadBuffer_` chunks and submits them to `codecStreamBridge_` once `kLocalVoiceChunkSamples` is reached. For this phase, voice export should reuse those dry mono chunks but only after the export session has crossed its chosen boundary-aligned start.

### Remote interval tap should happen after decode/resample, before mixer gain/pan
Decoded remote interval audio arrives back in the processor as `decodedAtHostRate` and then passes through `queueDecodedRemoteInterval(...)`. That is the right stem tap because it is already decoded and normalized to host sample rate, but not yet mixed or post-processed.

### Remote voice tap should happen after decode/resample, before playback queueing
Remote voice chunks also re-enter as `decodedAtHostRate` and then flow into `queueDecodedRemoteVoiceChunk(...)`. Export should capture those decoded chunks directly rather than tapping post-mix output.

### UI/settings seam is already processor-owned
The editor already works through processor-owned settings/state snapshots and `SessionSettingsSerializer`. This phase can extend that model with a small recording export state rather than inventing editor-only folder/arming behavior.

## Architecture Patterns

### Pattern 1: Processor Enqueues, Writer Thread Owns Files
The processor should never touch `juce::FileOutputStream`, `juce::WavAudioFormat`, or create directories inside `processBlock()`. It should enqueue immutable work packets to a writer thread that owns:
- session folder creation
- per-source file creation
- WAV headers/writers
- chunk appends
- clean close/finalization on stop/disconnect/destruction

### Pattern 2: Boundary-Gated Recording State
Recording armed does not mean “write immediately.” The phase needs explicit state like:
- not armed
- armed / waiting for next boundary
- actively recording
- stopping at next boundary

That state should be processor-owned so local interval, local voice, and remote interval paths all agree on when capture is allowed to begin or end.

### Pattern 3: Per-Source Export Streams With Stable Identity
Remote stems should be keyed by a stable source identity, not by mixer row position. The current code already uses source ids and `RemoteSourceInfo`; export should continue that model, then derive filenames from username + channel name at the writer edge.

### Pattern 4: Tap Audio Before Post-Mix Controls
Per-strip gain/pan/mute/solo and master output belong to monitoring, not to exported stems. Export should therefore tap:
- local interval: `localUploadIntervalBuffer_` on submit
- local voice: `localVoiceUploadBuffer_` on chunk submit
- remote interval: `decodedAtHostRate` before queueing
- remote voice: `decodedAtHostRate` before queueing

### Pattern 5: Reconnect Continues Session Folder, Not File Stream
The user wants reconnect to stay in the same session folder. The safest interpretation is:
- keep the same top-level recording session folder
- close/finalize any open files on disconnect
- after reconnect, lazily create new files for the first eligible boundary/audio event

This avoids pretending that audio continuity survived a disconnect.

### Anti-Patterns to Avoid
- Writing files directly from `processBlock()`.
- Recording post-strip or post-master output instead of dry source audio.
- Starting voice export immediately on toggle rather than honoring the boundary-aligned start rule.
- Backfilling silence for late joiners after the user explicitly ruled it out.
- Coupling exported file identity to volatile UI labels or strip order.

## Common Pitfalls

### Pitfall 1: Boundary start implemented only for interval mode
Voice is continuous, but the user still wants voice export to begin on the next interval boundary. If we only gate interval stems, voice files will violate the locked workflow.

### Pitfall 2: File/folder creation on first audio callback
Even one lazy `createDirectory()` or `createWriterFor()` on the audio thread can destabilize the session. File lifecycle must be deferred to the background service.

### Pitfall 3: Reconnect writes into stale open file handles
If disconnect does not finalize writer state cleanly, reconnect can produce corrupt WAV headers or misleading contiguous files.

### Pitfall 4: Name collisions with repeated users/channels
Public rooms can have repeated channel names. The writer service should resolve filenames once and own numeric suffixing centrally.

## Recommended File/Module Shape

```text
src/
+-- app/session/StemCaptureSettings.*          # small POD / validation for UI-facing export settings
+-- audio/StemCaptureWriter.*                  # background worker service and WAV file/session ownership
`-- plugin/FamaLamaJamAudioProcessor.*         # boundary gating, enqueue points, reconnect lifecycle

tests/
+-- unit/stem_capture_writer_tests.cpp
`-- integration/plugin_stem_capture_tests.cpp
```

This keeps file IO and path logic out of the plugin editor and lets the processor focus on timing/capture policy.
