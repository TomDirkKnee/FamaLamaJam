## Summary

The current codebase is single-bus by construction, not just by accident. The processor constructor declares exactly one stereo input and one stereo output, `isBusesLayoutSupported()` only accepts stereo main in/stereo main out, local transmit capture reads from the aggregate `processBlock()` buffer, and remote audio is always summed into the same main output buffer. That means any routing phase that adds more inputs or outputs must first separate bus ownership inside `processBlock()`; it is not enough to add UI or route tables on top of the current buffer assumptions.

JUCE and VST3 do support multiple named buses, and JUCE explicitly provides `BusesProperties`, `getBusBuffer()`, and bus-layout callbacks for this model. However, JUCE also documents that dynamic bus add/remove is uncommon and disabled by default, and VST3 makes bus activation host-controlled rather than plugin-controlled. Ableton Live 12 officially documents two host-side workflows that matter here: plug-in sidechain routing for extra inputs, and track-routing access to multi-I/O plug-ins, including up to 16 stereo outputs. What Ableton documents clearly is one sidechain-style extra input and multi-output routing in general; what it does not document clearly is arbitrary multiple aux inputs for third-party effects or the exact UX of extra output pairs for an audio-effect instance on an audio track. That is why the safest planning target is a fixed-bus proof, not a dynamic routing matrix.

**Primary recommendation:** Plan a narrow proof phase that adds one stereo aux input bus and one stereo aux output pair, rewrites the processor to use per-bus buffers via `getBusBuffer()`, validates Ableton sidechain input plus routed sub-output behavior manually, and defers dynamic bus counts plus larger routing UX until that proof passes.

### Pattern 1: Fixed Declared Buses, Host-Activated Usage
**What:** Declare a small fixed set of named stereo buses in the processor constructor, and let the host activate or ignore the non-main buses.
**When to use:** First proof phase and likely first shipping version of routing.
**Why:** JUCE and VST3 both support this directly, while dynamic bus growth is host-sensitive and not the right first commitment.
**Example:**
```cpp
FamaLamaJamAudioProcessor::FamaLamaJamAudioProcessor()
    : juce::AudioProcessor(juce::AudioProcessor::BusesProperties()
          .withInput("Main In", juce::AudioChannelSet::stereo(), true)
          .withInput("Local Send 2", juce::AudioChannelSet::stereo(), false)
          .withOutput("Main Out", juce::AudioChannelSet::stereo(), true)
          .withOutput("Remote Out 1", juce::AudioChannelSet::stereo(), false))
{
}
```
Source: https://docs.juce.com/master/classjuce_1_1AudioProcessor.html

### Pattern 2: Per-Bus `processBlock()` Access
**What:** Treat the aggregate process buffer as a container of bus buffers, not as one flat stereo stream.
**When to use:** Immediately once an extra input or output bus exists.
**Why:** JUCE explicitly requires `getBusBuffer()` or `getChannelIndexInProcessBlockBuffer()` once more than one bus exists, and warns that surplus output channels may contain garbage if not overwritten.
**Example:**
```cpp
void FamaLamaJamAudioProcessor::processBlock(juce::AudioBuffer<float>& buffer, juce::MidiBuffer&)
{
    auto mainIn = getBusBuffer(buffer, true, 0);
    auto mainOut = getBusBuffer(buffer, false, 0);
    auto extraIn = getBusBuffer(buffer, true, 1);
    auto routedOut = getBusBuffer(buffer, false, 1);

    routedOut.clear();

    // Main monitor/mix path still targets mainOut.
    // Extra local transmit reads only from extraIn.
    // Routed remote sources are written only to routedOut.
}
```
Source: https://docs.juce.com/master/classjuce_1_1AudioProcessor.html

### Pattern 3: Post-Decode Output Projection
**What:** Keep decode, timing, and remote-source identity unchanged; add a small route-assignment layer that decides whether a decoded source is written to main output or a routed aux output.
**When to use:** Output-side proof and follow-on implementation.
**Why:** FLJ already tracks `sourceId=user#channelIndex` and `groupId=user`, so routing can be layered after decode without reopening transport semantics.
**Example:**
```cpp
struct OutputRoute
{
    std::string sourceId;
    int outputBusIndex;
};

// Default: source not in route table -> main output.
// Proof: one selected remote source -> output bus 1.
```
Source: local code inspection of `src/net/FramedSocketTransport.cpp:1381-1388` and `src/plugin/FamaLamaJamAudioProcessor.cpp:1096-1128`

### Minimal Proof Path
1. Add one stereo aux input bus named for a second local transmit path.
2. Add one stereo aux output bus pair named for one routed remote return.
3. Update `isBusesLayoutSupported()` so main I/O remains stereo and the extra buses, when present, are also stereo.
4. Replace aggregate-buffer assumptions in `processBlock()` with `getBusBuffer()` access for main input, aux input, main output, and routed output.
5. Feed the aux input bus into a second local upload accumulator while keeping current local monitor behavior on the main bus.
6. Add a tiny route table that maps exactly one selected decoded remote source to the extra output bus; all other sources continue to the main output.
7. Validate in Ableton that the extra input is available through the plug-in sidechain UI and the extra output pair can be selected from another track's `Audio From` chooser.

### What Should Stay Research / Documentation
- Arbitrary runtime add/remove local input buses.
- Unlimited per-remote-channel host output buses.
- Full routing-matrix UI inside FLJ.
- Any plan that assumes FLJ can browse Ableton's track list directly from the plugin.

### What Is Safe Proof-of-Concept Work
- One extra stereo aux input bus.
- One extra stereo aux output pair.
- One fixed remote-source-to-output-bus route.
- Processor integration tests for bus separation and routed-output correctness.
- Manual Ableton validation notes that become the basis for a follow-on `VALIDATION.md`.

### Anti-Patterns to Avoid
- **Flat-buffer assumptions after adding buses:** The current code reads and writes the aggregate buffer directly; this will corrupt semantics once aux buses exist.
- **Dynamic buses in the first proof:** JUCE defaults to no dynamic add/remove support, and host behavior is too variable to make that the first milestone.
- **Treating sidechain as audible monitor audio:** Ableton documents sidechain as a trigger/input path, not an automatically heard second signal path.
- **Leaving aux outputs uncleared:** JUCE explicitly warns that output-only channels may contain garbage unless overwritten.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bus-to-buffer channel math | Custom absolute-channel indexing logic everywhere | `getBusBuffer()` and `getChannelIndexInProcessBlockBuffer()` | JUCE already owns the mapping and warns about host-specific buffer layout details |
| Dynamic routing topology discovery | Runtime add/remove bus system as the first implementation | Fixed declared buses plus host activation | Better match for JUCE defaults and safer host behavior |
| Ableton track enumeration inside the plugin | Custom host-session browser assumptions | Host routing and sidechain choosers | Reviewed official docs support routing through Live, not plugin-owned track browsing |
| Output grouping identity | New remote identity layer | Existing `sourceId=user#channelIndex` and `groupId=user` | The current transport already distinguishes per-channel and per-user identities |
| Host output breakout UX | Custom FLJ patchbay-first design | Ableton track routing from plugin outputs | This is the documented host workflow users already understand |

**Key insight:** The hard part here is not transport or codec work. The hard part is host bus semantics and safe processor buffer ownership. Reuse JUCE/VST3 bus machinery and Ableton's routing UX instead of inventing a plugin-local patchbay too early.

## Common Pitfalls

### Pitfall 1: Assuming Bus Declaration Equals Bus Availability
**What goes wrong:** The plugin declares an aux bus and the plan assumes the host will always activate and present it.
**Why it happens:** VST3 bus activation is host-controlled, and even default-active is only a hint.
**How to avoid:** Treat non-main buses as optional at runtime and validate actual host behavior in Ableton before expanding scope.
**Warning signs:** The bus exists in code but no sidechain control or sub-output choice appears in Live.

### Pitfall 2: Reusing the Main `processBlock()` Buffer as if Nothing Changed
**What goes wrong:** Local send 2 or routed output audio bleeds into the wrong path because code still reads/writes the aggregate buffer as a single stream.
**Why it happens:** The current codebase is built around one stereo in/out assumption.
**How to avoid:** Move immediately to named per-bus buffer access and clear/write aux outputs explicitly.
**Warning signs:** Extra outputs contain copies of main input, stale data, or partial sums.

### Pitfall 3: Overplanning Around Multiple Ableton Aux Inputs
**What goes wrong:** The plan assumes an arbitrary number of user-routable extra inputs for an audio effect.
**Why it happens:** JUCE/VST3 make multiple inputs look straightforward, but Ableton documentation clearly covers sidechain-style input and does not clearly document multiple independently user-routable aux inputs for this case.
**How to avoid:** Treat one extra stereo input as the proof target and keep larger-N input expansion contingent on manual Live evidence.
**Warning signs:** Planning language starts promising N local sends without a tested host workflow.

### Pitfall 4: Confusing Per-Channel Logic With Unlimited Per-Channel Buses
**What goes wrong:** The implementation exposes too many host outputs because FLJ already has per-channel identities internally.
**Why it happens:** Internal routing identity is not the same thing as practical host I/O ergonomics.
**How to avoid:** Keep per-channel logical assignment, but cap exposed routed bus pairs and fall back to per-user grouping if needed.
**Warning signs:** Route plans scale with room channel count instead of a small fixed host bus budget.

### Pitfall 5: Moving Decode/Timing Work Into the Routing Layer
**What goes wrong:** The output-routing phase reopens remote transport semantics and timing code unnecessarily.
**Why it happens:** Routing is being tackled before respecting the current post-decode architecture.
**How to avoid:** Keep routing downstream of decode and boundary alignment. Only change where already-decoded audio is written.
**Warning signs:** Proof work touches `FramedSocketTransport` timing semantics instead of processor bus/output code.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Treat the plugin as one flat stereo buffer | Treat the plugin as a set of named buses accessed by `getBusBuffer()` | Current JUCE/VST3 bus model | Required for safe multi-I/O processing |

**Deprecated / outdated planning assumptions:**
- Dynamic bus growth as the first milestone: outdated for this project's risk profile.
- Aggregate main-buffer processing after adding aux buses: invalid once more than one bus exists.
- Planning around unlimited per-channel host outputs: too optimistic without a verified Live workflow.

## Open Questions

1. **How does Ableton present extra output pairs for this specific audio-effect workflow?**
What we know: Live 12 documents that some plug-ins expose multiple I/O features through track routing, and Ableton support documents access to up to 16 stereo outputs.
What's unclear: The support article demonstrates the workflow on a MIDI track with a multi-out plug-in, not specifically a third-party audio effect on an audio track.
Recommendation: Keep output proof to one extra stereo pair and require manual Live confirmation before planning more.

2. **Does Ableton expose more than one practical extra input path for a third-party effect?**
What we know: Live 12 clearly documents a sidechain-capable plug-in input path.
What's unclear: Official Ableton documentation reviewed here does not clearly document multiple independently user-routable aux inputs for this case.
Recommendation: Treat one extra stereo input as HIGH-priority proof work and keep larger-N input expansion as research-only until manually verified.

3. **How many routed output pairs should FLJ expose if the proof succeeds?**
What we know: FLJ already has per-channel identities and Ableton documents up to 16 stereo outputs for multi-out plug-ins.
What's unclear: The practical UX ceiling for an audio-effect workflow in real Ableton sessions.
Recommendation: Plan the follow-on implementation around a small fixed cap, then fall back to per-user grouping if per-channel bus count becomes clumsy.
