# Phase 08.3.4 Plan 02: Host Multi-I/O Routing Research Summary

**Fixed-bus processor proof with one aux local-send input, one routed remote output pair, and focused routing coverage**

## Performance

- **Duration:** 23 min
- **Started:** 2026-04-04T20:04:00Z
- **Completed:** 2026-04-04T20:27:01Z
- **Tasks:** 2
- **Files modified:** 4

## Accomplishments

- Declared `Local Send 2` and `Remote Out 1` as fixed proof buses, enabled them in the live layout, and narrowed bus-layout acceptance to stereo main I/O plus stereo-or-disabled proof buses.
- Reworked `processBlock()` around named bus views so the main monitor path stays on the main output, aux-input activity is captured separately for proof diagnostics, and the aux output is explicitly cleared each block.
- Routed one selected decoded remote `sourceId` to the proof output pair after decode while leaving unassigned sources on the main output, and proved the same rule holds for voice-mode chunks.

## Files Created/Modified

- `src/plugin/FamaLamaJamAudioProcessor.h` - Adds the proof-route assignment seam alongside the fixed-bus routing snapshot.
- `src/plugin/FamaLamaJamAudioProcessor.cpp` - Declares the proof buses, switches processing to per-bus views, and routes one selected decoded source to the aux output pair.
- `tests/integration/plugin_host_multi_io_routing_tests.cpp` - Verifies bus declaration, stereo layout gating, aux-input isolation, routed-output isolation, aux clearing, and voice-mode routing.
- `tests/integration/plugin_voice_mode_transport_tests.cpp` - Updates existing voice-mode verification to read the processor's main output bus under the widened bus layout.

## Decisions Made

- Keep the proof buses exposed by default so Ableton can actually see the fixed stereo declaration during the later host pass, but still accept disabled aux layouts at runtime.
- Treat the routed-output proof as a one-entry route table only: output bus 1 can carry one selected `sourceId`, and everything else remains on the main mix.
- Preserve compatibility with older two-channel test harnesses inside `processBlock()` rather than rewriting unrelated suites before the routing proof is finished.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking Issue] Reworked the RED harness to use bus views and short deterministic timing**
- **Found during:** Task 1 (Add the extra stereo input bus and separate per-bus processing)
- **Issue:** The existing proof tests wrote aggregate channels directly and used the mini-server's default 16-beat interval, which never activated the routed interval within the test timeout once the processor moved to real bus ownership.
- **Fix:** Updated the routing tests to fill and inspect named bus buffers with `getBusBuffer()` and switched the routed-output proof runs to a short `400 BPM / 1 BPI` timing window.
- **Files modified:** `tests/integration/plugin_host_multi_io_routing_tests.cpp`
- **Verification:** `build-vs-2026/tests/Debug/famalamajam_tests.exe "[plugin_host_multi_io_routing]"`
- **Committed in:** `6964852`

**2. [Rule 1 - Bug] Added an undersized-buffer compatibility path after the processor widened to four channels**
- **Found during:** Task 2 (Route one selected decoded remote source to the extra output pair)
- **Issue:** Existing mixed-room voice transport tests still supplied two-channel aggregate buffers, which no longer matched the processor's declared multi-bus layout and broke verification.
- **Fix:** Hardened `processBlock()` to fall back to the main bus when a legacy harness passes an undersized aggregate buffer, and updated the voice transport tests to inspect the main output bus explicitly under the widened layout.
- **Files modified:** `src/plugin/FamaLamaJamAudioProcessor.cpp`, `tests/integration/plugin_voice_mode_transport_tests.cpp`
- **Verification:** `build-vs-2026/tests/Debug/famalamajam_tests.exe "[plugin_host_multi_io_routing],[plugin_voice_mode_transport]"`
- **Committed in:** `0a15360`
