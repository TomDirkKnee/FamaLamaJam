## Phase 08.3.4 Context

### Goal

Research and validate what JUCE, VST3, and Ableton can realistically support for:
- multiple local transmit inputs into FamaLamaJam
- routing decoded remote channels to distinct plugin output pairs

The phase is explicitly research-first. It should document what is practical before the project commits to a larger routing implementation.

### Current Code Reality

- The plugin currently exposes exactly one stereo input bus and one stereo output bus.
- `isBusesLayoutSupported()` currently only accepts stereo main input and stereo main output.
- Local transmit capture currently reads directly from the single `processBlock()` input buffer.
- Remote interval and voice audio are already decoded before they reach the mixer/output stage, so output routing should be treated as post-decode routing rather than mode-specific transport behavior.

### Input-Side Decisions

- Split the phase mentally into input research first, then output research.
- Primary research target: multiple plugin input buses.
- One stereo Ableton input bus should map to one NINJAM transmit channel.
- Initial proof target is one extra local transmit channel beyond the current local monitor.
- Longer-term desired direction is arbitrary add/remove local transmit channels dynamically.
- Local channel naming should default from host/bus naming when available, but remain user-editable inside FLJ.

### Output-Side Decisions

- Remote routing target is exposed FLJ output bus pairs, not direct Ableton track enumeration from inside the plugin UI.
- Default route should remain the FLJ main output.
- Output assignment should happen after decode and be mode-agnostic:
  - once a remote channel becomes playable audio inside FLJ, interval versus voice should not matter to routing behavior
- Preferred routing granularity is one stereo output pair per remote channel.
- Initial proof target can be smaller than the full dynamic model, but the research should judge whether per-channel routing is practical enough to pursue.
- If full per-remote-channel routing proves impractical, the preferred fallback is one stereo output pair per remote user.

### What This Phase Must Answer

1. Can FLJ expose additional meaningful host input buses in a way Ableton users can actually route into?
2. Can one extra routed local transmit channel be proven end-to-end before scaling further?
3. Can FLJ expose additional output bus pairs that Ableton can use for remote-channel routing?
4. Is per-remote-channel output routing practical enough to recommend, or should the project fall back to per-user routing?
5. Should any resulting implementation become its own follow-on execution phase, or remain out of scope because host ergonomics are too weak?

### Explicit Non-Goals

- Do not implement the full dynamic multi-I/O system in this phase unless the research plan deliberately chooses a very small proof-of-concept.
- Do not redesign the mixer layout here; Phase 9 remains the place for the bigger horizontal mixer/UI refresh.
- Do not assume FLJ can enumerate arbitrary Ableton session tracks directly in plugin UI dropdowns unless research proves that explicitly.
