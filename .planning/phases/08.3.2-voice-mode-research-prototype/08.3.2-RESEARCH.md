## Summary

This phase should be planned as a true protocol and runtime behavior change, not as a cosmetic follow-on to the existing unsupported-voice guard. FLJ already carries enough metadata to identify voice channels, but the current pipeline is interval-centric end to end: local upload batches full intervals, remote playback assumes completed interval payloads, and the processor promotes decoded audio only on interval boundaries. JamTaba's implementation shows that real NINJAM voice mode is different on both transmit and receive: small chunks are flushed immediately, encoder settings are lower quality, and playback promotes chunk decoders without waiting for interval boundaries.

The codebase already has the right broad seams for this work. `FramedSocketTransport` owns per-channel metadata and outbound framing, `CodecStreamBridge` already separates input submission from outbound chunk retrieval and inbound decode delivery, and the processor/editor already surface strip mode and transmit-state information. The main planning risk is mixing voice mode into the existing interval scheduler too late in the pipeline. If voice chunks are allowed to flow into the interval queueing path, FLJ will repeat the earlier compatibility problem instead of delivering true near-realtime voice.

The safest plan shape is:
- Wave 1: extend transport, codec, harness, and processor seams so voice frames can be sent and received as chunked mono audio with automated mixed-room coverage.
- Wave 2: wire local toggle behavior and mixed-room UI semantics into the existing strip model.
- Wave 3: run a focused automated gate, rebuild the debug VST3, and validate mixed interval-plus-voice behavior manually in Ableton.

## Architecture Patterns

### Pattern 1: Processor-Owned Strip State, Editor as Projection
Keep the local voice toggle state, transient switch text, remote voice strip status, and orange/green/red semantics in processor-exported strip snapshots. The editor should render this state, not invent it.

### Pattern 2: Split Interval and Voice Audio Semantics Early
Voice-mode payloads must branch before interval buffering and boundary scheduling. The transport or processor can still share common metadata and strip bookkeeping, but voice chunks should not enter the completed-interval queue path.

### Pattern 3: Mixed-Room Validation Through Harness Flags
The protocol seam is already there through `channelFlags`; the plan should extend tests so one remote peer can advertise interval mode while another advertises voice mode in the same room and both remain valid.

### Pattern 4: Live Mode Switching With Explicit Transitional State
The user wants mode changes to apply live. That means the phase should model transition states explicitly rather than pretending a mode flip is instantaneous and invisible.

### Anti-Patterns to Avoid
- Treating voice frames as interval audio and only fixing it in the render loop.
- Building a separate "voice chat" UI path that duplicates strip controls.
- Hiding quiet voice peers instead of keeping their strip visible.
- Overloading the existing transmit button into a multi-mode cycling control.
- Pulling Phase 9 mixer redesign work into the prototype.

## Common Pitfalls

### Pitfall 1: Voice Receive Gated by Interval Boundaries
If voice payloads still wait for interval promotion, playback will not be near-realtime and mixed-room semantics will feel broken.

### Pitfall 2: Mode Changes Without Encoder Reset
JamTaba recreates encoder behavior when a channel flips into voice mode. Reusing interval encoder assumptions may keep latency too high or quality settings wrong.

### Pitfall 3: Ambiguous Local Strip Truth
If the strip only changes color but not wording, users may not know whether they are transmitting voice, merely armed for voice, or still warming back into interval mode.

### Pitfall 4: No Automated Mixed-Room Coverage
Manual-only validation would be too weak here. The phase needs harness-level proof that interval and voice peers can coexist without the older unsupported-voice failure mode.
