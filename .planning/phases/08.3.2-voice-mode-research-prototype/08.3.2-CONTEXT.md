# Phase 08.3.2: Voice Mode Research & Prototype - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Research and prototype true NINJAM voice-chat mode as a per-channel behavior, using JamTaba as the reference for rooms where some channels remain interval-based and others use near-realtime voice passthrough. This phase covers the local voice toggle, mixed-room send/receive semantics, and a practical first-pass quality/latency target.

This phase does not include stem capture/export, host multi-I/O routing, or the broader horizontal mixer/layout redesign. Those remain later phases.

</domain>

<decisions>
## Implementation Decisions

### Mixed-room voice model
- Voice mode is a per-channel behavior, not a room-wide switch.
- Remote voice-mode channels should stay in the normal mixer, using the same strip and controls as interval channels.
- Remote voice strips should be clearly marked as voice mode with orange styling and explicit explanatory text.
- When the local strip is switched into voice mode, interval upload for that strip stops; FLJ should not send interval and voice payloads in parallel for the same local strip.
- Mode changes should apply live during a connected session rather than requiring reconnect.
- Remote voice-mode audio should still use the normal strip gain/pan/mute/solo path rather than a separate monitoring subsystem.

### Local toggle behavior
- The local voice toggle should be a separate control beside the existing transmit button, not merged into a single cycling control.
- If voice mode is enabled locally while transmit is off, the local strip should read `Voice mode ready`.
- When switching from interval mode into voice mode while transmitting, the strip should briefly show `Switching to voice mode...`.
- When switching back from voice mode to interval mode while connected, FLJ should return through the normal interval warmup path before going green again.
- The existing transmit truth remains explicit:
  - red = not transmitting
  - green = interval transmit
  - orange = voice mode

### Quality and latency target
- Phase 08.3.2 should treat voice mode as conversation-first, not music-first.
- The first prototype should be mono, not stereo.
- The UI should show a short inline hint on the local strip explaining the tradeoff, such as `Voice chat: low quality, near realtime`.
- If a remote voice source goes quiet temporarily, the strip should remain present and the audio path should simply go quiet rather than hiding the strip or introducing a separate idle-state model for voice.

### Claude's Discretion
- Exact wording of the orange voice-mode explanatory text, as long as it stays explicit and short.
- Exact iconography or badge styling for the local voice toggle.
- Exact timing of the transient `Switching to voice mode...` message, as long as the switch remains understandable to the user.
- Exact low-quality voice encoder settings for the first prototype, as long as they align with the conversation-first goal and JamTaba-style semantics.

</decisions>

<specifics>
## Specific Ideas

- The user wants FLJ voice mode to behave like JamTaba: some people can stay in delayed interval mode while others talk in near-realtime voice mode in the same room.
- The mixer should not split into a separate voice-chat area; the same strip model should remain intact.
- Orange is the reserved color for voice mode so it is visually distinct from green interval transmit and red not-transmitting states.
- The prototype should optimize for quick communication, not for high-fidelity music passthrough.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- FLJ already carries per-channel `channelFlags` through `FramedSocketTransport`, so the transport metadata path needed for voice mode already exists.
- The local strip already has explicit transmit-state UI with red/amber/green semantics and can host an adjacent voice toggle rather than inventing a separate surface.
- FLJ currently has a remote voice-mode compatibility guard, including `unsupportedVoiceMode` state in mixer snapshots and editor styling hooks, which can be evolved into true support rather than replaced.
- Diagnostics already expose `mode=voice` and source-level receive state, which can help validate the prototype.

### JamTaba Reference Behavior
- JamTaba encodes voice mode per channel flag (`2`) rather than as a room-wide mode.
- JamTaba lowers the send threshold dramatically for voice channels so small chunks are sent immediately instead of waiting for interval-sized payloads.
- JamTaba recreates the encoder for a channel when voice mode changes and uses lower encoding quality for voice chat.
- JamTaba treats voice-mode receive/playback as chunked near-realtime audio and promotes the next decoder immediately instead of waiting for interval boundaries.

### Existing Constraints
- FLJ currently treats remote voice-mode channels as unsupported and deliberately avoids mixing them as interval audio.
- The current plugin remains a single stereo input and single stereo output bus, so Phase 08.3.2 should not assume multi-input routing or per-stream DAW outputs.
- The broader horizontal mixer redesign remains deferred to Phase 9, so this prototype should fit inside the current single-page mixer/session UI.

### Integration Points
- Transport and per-channel flag handling live in `src/net/FramedSocketTransport.*`.
- Current send/receive codec and interval queue behavior live in `src/audio/CodecStreamBridge.*` and `src/plugin/FamaLamaJamAudioProcessor.cpp`.
- Local strip controls and color/state presentation live in `src/plugin/FamaLamaJamAudioProcessorEditor.cpp`.
- Existing remote voice guard tests in `tests/integration/plugin_voice_mode_guard_tests.cpp` should be a likely starting point for the prototype test seam.

</code_context>
