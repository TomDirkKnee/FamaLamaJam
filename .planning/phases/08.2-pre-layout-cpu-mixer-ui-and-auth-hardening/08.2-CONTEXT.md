# Phase 08.2: Pre-Layout CPU, Mixer, UI, and Auth Hardening - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Resolve the newly surfaced functional issues that currently make the plugin hard to rehearse with before the broader JamTaba-inspired layout work in Phase 9. This phase covers a practical interim layout fix, mixer cleanup around the dead default controls, password/auth workflow completion, and diagnosis-first CPU spike hardening. It does not attempt the full horizontal-strip redesign, solo support, or a broad visual overhaul.

</domain>

<decisions>
## Implementation Decisions

### Interim layout shape
- Phase 08.2 should move the room/chat workflow into a dedicated right-hand sidebar instead of leaving it above the mixer.
- The sidebar should use a sensible fixed width in this phase rather than adding a resizable layout system.
- The current mixer strip model should remain in place for now; making it visible and usable again is the goal, not starting the full Phase 9 mixer redesign early.
- The right-hand room area should stay visible even when disconnected so the layout does not jump around between states.
- The room feed should keep its own independent scroll area inside the sidebar.
- The room sidebar should continue to show topic, feed, composer, and BPM/BPI vote controls.
- BPM/BPI vote controls should remain near the top of the room sidebar, but be smaller and more inline than they are now.

### Mixer cleanup and master output
- The old `Default Gain`, `Default Pan`, and `Default Muted` controls should be removed from the top settings area.
- New mixer strips should start from simple unity defaults: `0 dB`, centered pan, and unmuted.
- A new master output control should be added to the Mixer section.
- That master output should affect the full final plugin output, including local monitor, delayed remote returns, and metronome.
- The master output should appear as a mixer footer control rather than a full extra strip.
- The existing Local Monitor strip should remain separate and keep its own identity; it should not be merged into the new master output.

### Password and auth workflow
- The connection UI should expose a real password field alongside Host, Port, and Username.
- The password field should remain visible all the time, even though public rooms may leave it blank.
- Password/auth failures should be shown inline and explicitly in the connection area rather than hidden behind vague generic status wording.
- Passwords should be saved in project state for private-room reconnect convenience.
- Phase 08.2 should include real validation of the password flow against a local or otherwise controlled private Ninjam server path, not just UI wiring or fake-server assumptions.

### CPU spike target
- Phase 08.2 should treat CPU work as diagnosis-first rather than promising JamTaba CPU parity up front.
- The phase should first inspect and compare the current hot path against likely JamTaba behavior to decide whether the spikes are code-driven or mostly machine-specific.
- If the spikes are in the plugin path, the phase should materially reduce them.
- The main success condition is removing or clearly reducing the periodic spikes rather than chasing abstract average-CPU improvements first.
- Slightly less-smooth UI updates are acceptable if that materially reduces spikes.
- If a current feature needs to be simplified first, realtime meter refresh is the first acceptable tradeoff before touching core chat, sync, or audio behavior.

### Claude's Discretion
- Exact fixed width of the right-hand room sidebar.
- Exact styling and compact arrangement of the smaller inline BPM/BPI vote controls.
- Exact master-output control shape, as long as it reads as a mixer-footer output control rather than reviving the old dead defaults.
- Exact profiling and comparison method used to decide whether the spike pattern is code-driven or machine-specific.

</decisions>

<specifics>
## Specific Ideas

- The user explicitly wants the chat moved to the right-hand side because the current room area makes the mixer unusable.
- The user wants the mixer visible again now, without turning this urgent phase into the full Phase 9 layout overhaul.
- The user called out that JamTaba performs noticeably better on the same test machine, so CPU investigation should compare our hot path against likely JamTaba behavior where possible.
- The user is concerned the spike pattern may partly reflect the test machine, so the CPU work should distinguish diagnosis from assumptions.
- The user wants the password/auth flow to become testable against a real private-room path, not remain hidden behind transport-only support.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/plugin/FamaLamaJamAudioProcessorEditor.cpp`: already contains the current connection area, room section, mixer viewport, and the dead top-level default gain/pan/mute controls that need replacing.
- `src/plugin/FamaLamaJamAudioProcessor.cpp`: already owns mixer-strip state, meter updates, room UI state, and the realtime audio/mixing path where the periodic spikes are likely to be investigated.
- `src/net/FramedSocketTransport.cpp`: already implements password-aware Ninjam authentication and explicit auth failure reporting at the transport layer.
- `src/app/session/SessionSettings.h` and `src/app/session/SessionSettings.cpp`: currently store `serverHost`, `serverPort`, `username`, and the old default channel controls, so this is the obvious place to thread a password field and remove or replace dead defaults.

### Established Patterns
- Earlier phases deliberately kept the UI in one JUCE page and deferred the broad JamTaba-inspired layout overhaul, so Phase 08.2 should stay an interim hardening step rather than a full redesign.
- Phase 4 established per-strip gain/pan/mute plus a distinct Local Monitor strip as the real mixer model.
- Phase 7 established room feed, composer, and vote controls inside the current page, and also documented that the larger right-hand chat layout belongs to a later broader layout phase.
- Phase 08.1 proved discovery correctness and kept manual Host/Port entry as the source of truth, so auth UI changes should fit that same visible connection area.

### Integration Points
- The editor layout changes will center on the existing room and mixer viewports in `FamaLamaJamAudioProcessorEditor`.
- The new master output control will need a processor-owned mix/output state path rather than reviving the old default-strip-seeding settings flow.
- Password support must thread through `SessionSettings`, editor draft/apply flow, and the existing `FramedSocketTransport::start(... username, password)` path.
- CPU analysis will likely focus on the `processBlock()` mix/meter/timing loop plus related UI refresh cadence, because those are the existing places where beat/interval-cadenced work already occurs.

</code_context>
