## Decisions Made

- Used deterministic work counters instead of profiler timing assertions so the regression stays stable across developer machines.
- Treated the editor refresh path as the first mitigation target because it showed unconditional room-feed rebuilds and unchanged strip updates immediately, which aligns with the plan's explicit "reduce UI churn before audio semantics" allowance.
- Compared the outcome against likely JamTaba behavior by treating JamTaba's lower observed CPU on the same machine as evidence that unconditional UI churn is a code-driven difference, not something to blame on host timing alone.
