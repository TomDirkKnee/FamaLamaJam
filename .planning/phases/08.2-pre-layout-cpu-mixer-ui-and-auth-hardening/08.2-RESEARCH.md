## User Constraints (from CONTEXT.md)

### Locked Decisions
#### Interim layout shape
- Phase 08.2 should move the room/chat workflow into a dedicated right-hand sidebar instead of leaving it above the mixer.
- The sidebar should use a sensible fixed width in this phase rather than adding a resizable layout system.
- The current mixer strip model should remain in place for now; making it visible and usable again is the goal, not starting the full Phase 9 mixer redesign early.
- The right-hand room area should stay visible even when disconnected so the layout does not jump around between states.
- The room feed should keep its own independent scroll area inside the sidebar.
- The room sidebar should continue to show topic, feed, composer, and BPM/BPI vote controls.
- BPM/BPI vote controls should remain near the top of the room sidebar, but be smaller and more inline than they are now.

#### Mixer cleanup and master output
- The old `Default Gain`, `Default Pan`, and `Default Muted` controls should be removed from the top settings area.
- New mixer strips should start from simple unity defaults: `0 dB`, centered pan, and unmuted.
- A new master output control should be added to the Mixer section.
- That master output should affect the full final plugin output, including local monitor, delayed remote returns, and metronome.
- The master output should appear as a mixer footer control rather than a full extra strip.
- The existing Local Monitor strip should remain separate and keep its own identity; it should not be merged into the new master output.

#### Password and auth workflow
- The connection UI should expose a real password field alongside Host, Port, and Username.
- The password field should remain visible all the time, even though public rooms may leave it blank.
- Password/auth failures should be shown inline and explicitly in the connection area rather than hidden behind vague generic status wording.
- Passwords should be saved in project state for private-room reconnect convenience.
- Phase 08.2 should include real validation of the password flow against a local or otherwise controlled private Ninjam server path, not just UI wiring or fake-server assumptions.

#### CPU spike target
- Phase 08.2 should treat CPU work as diagnosis-first rather than promising JamTaba CPU parity up front.
- The phase should first inspect and compare the current hot path against likely JamTaba behavior to decide whether the spikes are code-driven or mostly machine-specific.
- If the spikes are in the plugin path, the phase should materially reduce them.
- The main success condition is removing or clearly reducing the periodic spikes rather than chasing abstract average-CPU improvements first.
- Slightly less-smooth UI updates are acceptable if that materially reduces spikes.
- If a current feature needs to be simplified first, realtime meter refresh is the first acceptable tradeoff before touching core chat, sync, or audio behavior.

## Summary

Phase 08.2 should be planned as four bounded workstreams that share existing seams instead of introducing new subsystems: a fixed-width right sidebar inside the current single-page JUCE editor, mixer cleanup that removes dead default controls and adds a true post-mix master output, end-to-end password threading through saved project state into `FramedSocketTransport`, and a diagnosis-first CPU pass focused on the realtime mix loop plus the editor's unconditional 20 Hz refresh path.

The repo already contains most of the architecture needed for this phase. The editor is a polling view over processor-owned state, the transport already supports NINJAM password auth and explicit failure reasons, mixer strips already use stable `sourceId` identities with persisted mix state, and non-audio room/discovery work is already drained outside `processBlock()`. The main planning risk is not missing infrastructure; it is changing the wrong layer or widening scope into Phase 9.

The clearest planning split is: first add the minimum state seams (`password`, `masterOutputGainDb`) and the tests that prove them, then land the fixed sidebar and mixer footer UI on top of those seams, and only then do the CPU pass with instrumentation so the phase can prove whether spikes come from audio-thread mixing, message-thread refresh churn, or both.

**Primary recommendation:** Plan Phase 08.2 around existing processor-owned state seams, with CPU work gated by measurement and UI/layout work kept strictly to an interim fixed two-column editor.

### Pattern 1: Processor-Owned State, Editor as Polling View
**What:** Keep connection settings, lifecycle copy, mixer state, room state, and output state in the processor. Let the editor read snapshots and write through narrow handlers.
**When to use:** Password UI, inline auth failure copy, right-sidebar state, master output footer control.
**Example:**
```cpp
// Source: src/plugin/FamaLamaJamAudioProcessor.cpp
auto settingsGetter = [this]() { return settingsStore_.getActiveSettings(); };
auto mixerStripSetter = [this](const std::string& sourceId, float gainDb, float pan, bool muted) {
    return setMixerStripMixState(sourceId, gainDb, pan, muted);
};

return new FamaLamaJamAudioProcessorEditor(*this,
                                           std::move(settingsGetter),
                                           std::move(apply),
                                           std::move(lifecycleGetter),
                                           std::move(connect),
                                           std::move(disconnect),
                                           std::move(transportUiGetter),
                                           std::move(hostSyncAssistUiGetter),
                                           std::move(hostSyncAssistToggle),
                                           std::move(mixerStripsGetter),
                                           std::move(mixerStripSetter),
                                           ...);
```

### Pattern 2: Keep Non-RT Room and Discovery Work Out of `processBlock()`
**What:** Room transport events, discovery results, and reconnect triggers already drain on a processor timer, not in the audio callback.
**When to use:** Auth error copy, room sidebar refresh logic, any new diagnostic counters that are not sample-accurate.
**Example:**
```cpp
// Source: src/plugin/FamaLamaJamAudioProcessor.cpp
void FamaLamaJamAudioProcessor::timerCallback()
{
    drainRoomTransportEvents();
    drainPublicServerDiscoveryResults();
    refreshPendingRoomVotesFromTiming();
    triggerScheduledReconnectForTesting();
}
```

### Pattern 3: Stable Mixer Identity by `sourceId`, Not Row Position
**What:** Mixer state is keyed by stable source IDs and persisted through the plugin state wrapper.
**When to use:** Preserve remote strip mix state after reconnect, add master output without disturbing existing strip identity rules.
**Example:**
```cpp
// Source: src/plugin/FamaLamaJamAudioProcessor.cpp
for (const auto& [sourceId, runtimeState] : mixerStripsBySourceId_)
{
    juce::ValueTree stripTree(kPluginStateMixerStrip);
    stripTree.setProperty(kPluginStateMixerStripSourceId,
                          juce::String(runtimeState.snapshot.descriptor.sourceId),
                          nullptr);
    stripTree.setProperty(kPluginStateMixerStripGainDb, runtimeState.snapshot.mix.gainDb, nullptr);
    stripTree.setProperty(kPluginStateMixerStripPan, runtimeState.snapshot.mix.pan, nullptr);
    stripTree.setProperty(kPluginStateMixerStripMuted, runtimeState.snapshot.mix.muted, nullptr);
    mixerStateTree.addChild(stripTree, -1, nullptr);
}
```

### Pattern 4: Master Output Is a Post-Mix Processor Stage
**What:** The current final buffer already combines local monitor, delayed remote returns, and metronome. The new master output must scale that combined result after those contributions are added.
**When to use:** Implementing the mixer footer output control.
**Example:**
```cpp

for (int sample = 0; sample < outputSamples; ++sample)
{
    // remote delayed returns are added here
    // metronome is added here
}

// Add master output gain after these contributions, not before.
```

### Anti-Patterns to Avoid
- **Reviving the dead default controls:** `defaultChannelGainDb`, `defaultChannelPan`, and `defaultChannelMuted` are currently only meaningful as strip-seeding values. Do not keep them as active top-level mixer UI.
- **Treating master output as another strip:** It must affect the final plugin output, including metronome and local monitor, so it cannot just be another `sourceId`.
- **Moving room/discovery parsing into `processBlock()`:** That reintroduces the exact realtime-path risk Phase 7 removed.
- **Starting Phase 9 early:** No resizable panes, tabs, popouts, horizontal strip redesign, or solo support in this phase.
- **Unconditional full UI rebuilds at 20 Hz:** `refreshRoomUi()` rebuilds feed widgets and `refreshMixerStrips()` walks every strip each tick. Reduce churn before adding more editor work if CPU diagnosis points there.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| NINJAM password auth | A new auth handshake implementation | Existing `FramedSocketTransport::start(... username, password)` and auth reply parsing | The transport already handles challenge hashing, auth timeout, and explicit failure text |
| Saved settings migration | A second serializer/blob format | `SessionSettingsSerializer` plus the wrapped plugin `ValueTree` state | Keeps fallback behavior and existing state restore tests valid |
| Mixer strip identity | Index-based UI state mapping | Existing `sourceId`-keyed `mixerStripsBySourceId_` snapshots | Reconnect/hide/show logic already depends on stable source identities |
| Private-room validation | Manual-only server testing | Extend `MiniNinjamServer` with configurable password acceptance/failure text | Gives repeatable auth success/failure coverage inside CI/local test runs |
| Layout framework for this phase | Resizable pane manager or popout workflow | Fixed-width sidebar in current editor `resized()` flow | Matches phase boundary and keeps Phase 9 work separate |

**Key insight:** This phase is mostly seam completion, not subsystem invention. The code already has the right ownership model.

## Common Pitfalls

### Pitfall 1: Adding `password` to the UI but Not to Saved State
**What goes wrong:** Private-room reconnect works until the project is saved/reloaded, then silently falls back to blank-password behavior.
**Why it happens:** Password must be threaded through `SessionSettings`, normalization/validation, serializer schema, editor draft/load, and `attemptLiveConnect()`.
**How to avoid:** Change the field end-to-end in one plan wave and add unit + roundtrip coverage before UI polish.
**Warning signs:** Manual connect works, but `plugin_state_roundtrip` still ignores the password field.

### Pitfall 2: Removing Default Controls Without Replacing Strip Seeding
**What goes wrong:** New remote or local strips stop coming up at unity defaults, or old saved projects restore unexpected per-strip values.
**Why it happens:** `ensureLocalMonitorMixerStrip()` and `syncRemoteMixerStrip()` currently seed from `defaultChannel*` session settings.
**How to avoid:** Replace those seeding reads with explicit unity defaults while keeping persisted per-strip state intact.
**Warning signs:** New strips appear muted/panned/gained incorrectly after the UI cleanup lands.

### Pitfall 3: Building Master Output as UI-Only Copy
**What goes wrong:** The footer slider moves, but metronome or delayed remote returns ignore it.
**Why it happens:** Current audio sources are added directly into the final output buffer inside `processBlock()`.
**How to avoid:** Add a processor-owned master gain stage that scales the final output buffer after local monitor, remote returns, and metronome have all been summed.
**Warning signs:** Local monitor changes but metronome loudness does not, or vice versa.

### Pitfall 4: Chasing CPU Spikes Blindly
**What goes wrong:** The phase spends time rewriting UI or DSP in the wrong place and still cannot explain the periodic spike pattern.
**Why it happens:** Two plausible hotspots already exist: the nested per-sample remote mixing loop and the editor's unconditional 20 Hz refresh/rebuild path.
**How to avoid:** Add cheap instrumentation first: per-block timing/counters around the remote mix loop, meter work, room feed rebuilds, and mixer refreshes. Compare with meter-refresh reductions before deeper changes.
**Warning signs:** Average CPU improves slightly, but every-4-beat spikes remain unchanged.

### Pitfall 5: Breaking Existing Layout Tests by Forgetting They Assert the Old Shape
**What goes wrong:** The right-sidebar refactor immediately fails UI coverage or, worse, stale tests keep passing while no longer checking the intended geometry.
**Why it happens:** Existing `plugin_room_controls_ui` and `plugin_rehearsal_ui_flow` tests explicitly assert that the room stays above the mixer.
**How to avoid:** Rewrite those assertions as part of the first layout wave, not after the UI code lands.
**Warning signs:** Tests still mention "above the mixer" after the sidebar branch exists.

### Pitfall 6: Relying on Lifecycle Copy Alone for Inline Auth Messaging
**What goes wrong:** Auth failures technically appear in the generic status line, but the connection area still does not clearly show password-related failure context next to the fields.
**Why it happens:** Current status plumbing is global, while the phase requirement is specifically "inline and explicitly in the connection area."
**How to avoid:** Reuse lifecycle error text, but render it in a dedicated connection-area label near Host/Port/Username/Password.
**Warning signs:** Failure text is visible somewhere in the page, but a user looking at the connection fields still has to hunt for it.

### Pitfall 7: Leaving `MiniNinjamServer` Auth Behavior Too Permissive
**What goes wrong:** The UI/state code passes a password, but automated tests still cannot prove wrong-password failure or right-password success.
**Why it happens:** The current test server accepts auth without validating the hashed password payload.
**How to avoid:** Extend the harness to validate or deliberately reject auth and emit configurable failure reply text.
**Warning signs:** SC4 still depends on manual private-room testing only.

## Open Questions

1. **Is the periodic spike primarily audio-thread work, editor refresh work, or both?**
   - What we know: `processBlock()` contains a nested per-sample/per-source remote mix loop, and the editor refreshes transport, room, and mixer UI at 20 Hz while rebuilding room feed widgets.
   - What's unclear: which side dominates on the user's test machine and whether the every-4-beat cadence tracks beat updates, interval work, or message-thread refresh.
   - Recommendation: make Wave 0 add instrumentation and one or two toggles (meter refresh reduction, room/mixer refresh throttling) before changing DSP behavior.

2. **Should master output persist with project state?**
   - What we know: the requirement asks for a real mixer footer control affecting final output, and current wrapped plugin state already persists metronome plus mixer strip state.
   - What's unclear: whether the user expects the master gain to reload with projects or reset to unity each session.
   - Recommendation: persist it in the wrapped plugin state, not `SessionSettings`, because it is mixer/output state rather than connection configuration.

3. **What exact auth failure strings should the controlled test server emit?**
   - What we know: `FramedSocketTransport` preserves explicit server failure text and falls back to `invalid login/password`.
   - What's unclear: which private-room server reply text the team wants to standardize around for tests and UI copy.
   - Recommendation: make the server harness configurable and assert against both explicit server text and the fallback path.
