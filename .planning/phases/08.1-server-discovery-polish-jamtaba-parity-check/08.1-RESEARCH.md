## Summary

Phase 8 proved the basic discovery workflow, but it also exposed two correctness gaps that are too user-visible to leave until Phase 9: the public-room counts are likely derived from the wrong source, and the picker preserves selection by row index rather than by endpoint identity. Those are both polish problems, but they affect trust in the room browser itself, not just the layout.

The current plugin fetches `http://ninjam.com/serverlist.php` in `src/infra/net/PublicServerDiscoveryClient.cpp` and parses text lines like `SERVER "host:port" "info" "1/8 users"` in `src/app/session/ServerDiscoveryModel.cpp`. Today we only extract the first integer from `usersText`, so `1/8 users` becomes `connectedUsers = 1`, and the UI still renders the raw string via `makePublicDiscoveryEntryLabel()` in `src/plugin/FamaLamaJamAudioProcessor.cpp`. That is simple, but it is also brittle and does not tell us whether the number reflects current non-bot users, all users including bots, or only a stale/max-capacity string.

JamTaba takes a different path. Its `LoginService` fetches JSON from `http://ninbot.com/app/servers.php` in `src/Common/loginserver/LoginService.cpp`, builds `RoomInfo` objects with an actual `users` array, and derives counts through `RoomInfo::getNonBotUsersCount()`. That explains why JamTaba can present room population more confidently and why it can distinguish current users from room capacity without scraping a text blob. This is the strongest evidence that our Phase 08.1 fix should be based on a richer discovery model rather than just prettier formatting of the existing raw `1/8` string.

The picker-jump bug is also clear in local code. `refreshServerDiscoveryUi()` in `src/plugin/FamaLamaJamAudioProcessorEditor.cpp` clears and rebuilds the combo box, then restores the previous selection using `getSelectedItemIndex()` / `setSelectedItemIndex(previousSelection, ...)`. When remembered entries are inserted or public entries are re-sorted, the same row index can point at a different server. That matches the user's report exactly.

**Primary recommendation:** implement a richer public-room discovery model informed by JamTaba's JSON login-service structure, sort public rooms by active user count descending while still grouping remembered entries first, and preserve picker selection by endpoint key rather than by row index. Keep the cramped layout issue explicitly deferred to Phase 9.

## Key Findings

### Finding 1: Current plugin discovery is text-scrape based
- `src/infra/net/PublicServerDiscoveryClient.cpp` fetches `http://ninjam.com/serverlist.php`.
- `src/app/session/ServerDiscoveryModel.cpp` parses `SERVER ... END` text and derives `connectedUsers` by taking the first integer found in `usersText`.
- `src/plugin/FamaLamaJamAudioProcessor.cpp` still shows the raw `usersText` string in labels when present.

### Finding 2: JamTaba uses a richer JSON login service
- `src/Common/loginserver/LoginService.cpp` fetches `http://ninbot.com/app/servers.php`.
- Each room carries `users`, `user_max`, `bpm`, `bpi`, and optional stream metadata.
- `RoomInfo::getNonBotUsersCount()` gives JamTaba a better definition of "how many users are really in the room."

### Finding 3: Public-room ordering is currently under-specified in our plugin
- We currently preserve remembered entries first, then public entries in feed order.
- The user's desired behavior is remembered-first grouping, with the public group sorted by active users descending.
- JamTaba's richer room data makes that ordering feasible without guessing from a raw `1/8` label.

### Finding 4: Picker selection drift is a state-key bug, not a transport bug
- The editor currently remembers the previously selected row index, not the selected endpoint.
- Any reordering after refresh or after a successful connect/remember operation can therefore make the combo box appear to "switch servers."

## Recommended Technical Direction

### Discovery model
- Add a richer parsed public-room model that can represent:
  - endpoint
  - room name/info
  - active user count
  - max users, when available
  - bpm / bpi, when available
- Prefer a JSON-capable discovery path compatible with JamTaba's login-service structure.
- Keep legacy text parsing only as a defensive fallback if needed.

### Ordering
- Continue grouping remembered entries first.
- Sort only the public section by active users descending.
- Break ties deterministically, for example by room name or endpoint key.

### Selection stability
- Track combo-box selection by endpoint key, not by row index.
- Re-apply selection after refresh/connect only if that endpoint still exists in the rebuilt list.
- If the endpoint no longer exists, leave the text fields untouched and let manual `Host` / `Port` remain the source of truth.

### Out of scope for 08.1
- Major discovery-area resizing, alternate layouts, or Phase 9 chat/mixer coexistence work.
- Broader visual redesign of row cards or a full JamTaba-like room browser panel.
