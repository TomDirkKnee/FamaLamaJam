**Confidence:** MEDIUM

## User Constraints (from CONTEXT.md)

### Locked Decisions
### Local strip model
- Extra local transmit channels should appear as full local strips, not compact subrows.
- Extra local transmit channels are user-created; they should not always exist by default.
- The user should create them through an inline `Add channel` control near the local-strip area.
- The default name should come from the host or bus name when available, otherwise `Local Send 2`.
- Local channel names are important and must be user-editable for meaningful instrument labels like `Bass` or `Guitar`.

### Per-channel controls
- `Transmit` becomes a global local-send control rather than a per-local-channel control.
- `Voice` becomes a global local-send mode rather than a per-local-channel control.
- When voice mode is enabled, all active local channels transmit voice.
- Each local channel still keeps its own gain, pan, mute, and solo controls.
- Removing an extra local channel should hide it but preserve its last settings so it can be restored easily within the session.

### Remote output routing UX
- Remote output assignment should live directly on each remote strip as a small dropdown.
- The dropdown should show explicit host-facing labels:
  - `FLJ Main Output`
  - `Remote Out 1`
  - `Remote Out 2`
  - and so on as supported
- If a remote channel is routed away from the main output, it should be removed from the FLJ main output completely rather than mirrored.

### Claude's Discretion
- Exact wording and visual style of the `Add channel` control.
- Exact inline editing affordance for local channel renaming.
- Exact placement and width of the remote output dropdown within the current pre-Phase-9 strip layout.
- Exact session-only versus project-persistent handling for hidden extra local channels, as long as the user-visible behavior matches "hide and remember".

### Deferred Ideas (OUT OF SCOPE)
- Arbitrary dynamic add/remove bus counts beyond the first `Main + Local Send 2` implementation remain a later expansion.
- Broader remote output fanout beyond the first fixed extra pairs remains later expansion work.
- Full mixer-density redesign, horizontal strips, embedded volume-in-meter controls, and broader layout changes remain Phase 9 work.
- Plugin-owned Ableton track enumeration remains out of scope; host-owned routing is the decided model.
</user_constraints>

## Summary

Phase `08.3.4` proved that Ableton exposes the fixed proof buses in usable host routing UI, and the current processor already owns the right low-level seam for bus splitting: constructor-declared buses, stereo-or-disabled layout gating, and `getBusBuffer()`-based routing in `processBlock()`. But the proof seam stops well short of a real multi-channel implementation. The current code still has one local sender model: one local strip identity, one local voice/transmit mode, one `CodecStreamBridge::submitInput()` queue, one `FramedSocketTransport::setLocalChannelInfo()` state, and one outbound upload path that hardcodes channel index `0` in `writeUploadInterval()`.

The planning-critical consequence is that this phase is not just a processor bus patch. It is a bounded multi-local-channel implementation across three layers: processor bus ownership, transport/codec metadata, and UI/state modeling. The processor already persists generic strip mix state by stable `sourceId`, which is useful, but the existing local-strip path is still special-cased enough that it will break the locked "hide and remember" behavior if extra local sends are modeled as copies of today's `LocalMonitor` strip. In particular, restore currently forces any `LocalMonitor` strip back to `active=true` and `visible=true`.

The safest implementation plan is a fixed-slot model. Keep the bus count compile-time fixed, with exactly two local input slots (`Main`, `Local Send 2`) and a small compile-time cap for routed remote output pairs. Replace `HostRoutingProof` with real fixed-cap state, split global local-send state from per-local-channel strip state, and extend the existing transport path so encoded outbound frames carry local channel index and channel metadata. In the editor, keep the UI minimal: global `Transmit` and `Voice` in a local-group header, an inline `Add channel` action that only reveals the one fixed extra local slot, full local strips for per-channel mix controls and naming, and a narrow dropdown on remote strips for output assignment.

**Primary recommendation:** Plan this phase around a fixed slot registry plus channel-indexed multi-local upload frames; do not keep extending `HostRoutingProof` ad hoc and do not reopen dynamic bus work.

### Pattern 1: Fixed Slot Registry, Not Dynamic Buses
**What:** Declare compile-time descriptors for local input slots and routed remote output buses.
**When to use:** Immediately in `08.3.4.1`.
**Why:** The proof already validated the host path for fixed buses. Planning around dynamic buses would add host sensitivity without solving a current user problem.
**Example:**
```cpp
struct LocalInputSlot
{
    const char* slotId;
    int inputBusIndex;
    const char* fallbackName;
    bool visibleByDefault;
};

static constexpr std::array<LocalInputSlot, 2> kLocalInputSlots {{
    { "local-main",   0, "Main",         true  },
    { "local-send-2", 1, "Local Send 2", false },
}};
```
Source: local code inspection of `src/plugin/FamaLamaJamAudioProcessor.cpp` and JUCE multi-bus constructor/docs: https://docs.juce.com/develop/classjuce_1_1AudioProcessor.html and https://docs.juce.com/master/structjuce_1_1AudioProcessor_1_1BusesProperties.html

### Pattern 2: Separate Global Local-Send State From Per-Channel Strip State
**What:** Keep `Transmit` and `Voice` in one processor-owned global state object, while each local slot owns only its own name, visibility, mix state, and meter.
**When to use:** For both processor snapshots and editor projection.
**Why:** Current code already stores `transmitEnabled_`, `localChannelMode_`, and `localVoiceTransitionPending_` as globals. Reusing `MixerStripSnapshot` as the canonical source of truth for global and per-strip state will keep producing duplicated UI rules.
**Example:**
```cpp
struct LocalSendGlobalState
{
    bool transmitEnabled { true };
    TransmitState transmitState { TransmitState::Disabled };
    LocalChannelMode mode { LocalChannelMode::Interval };
    bool voiceTransitionPending { false };
};

struct LocalChannelState
{
    std::string slotId;
    int inputBusIndex { 0 };
    std::string displayName;
    bool visible { false };
    MixerStripMixState mix;
    MixerStripMeter meter;
};
```
Source: local code inspection of `src/plugin/FamaLamaJamAudioProcessor.h` and `src/plugin/FamaLamaJamAudioProcessor.cpp`

### Pattern 3: Channel-Indexed Local Upload Frames
**What:** Extend the existing codec/transport path so each encoded outbound local frame carries channel index and channel metadata.
**When to use:** This phase's Wave 1 or Wave 2 implementation work.
**Why:** Today `CodecStreamBridge::submitInput()` carries only audio plus sample rate, `popEncoded()` returns only a payload, `FramedSocketTransport::setLocalChannelInfo()` stores one channel name/flag set, and `writeUploadInterval()` hardcodes channel index `0`. Without changing that, the server can only see one local channel no matter how many host buses FLJ reads.
**Example:**
```cpp
struct EncodedLocalFrame
{
    juce::MemoryBlock payload;
    std::uint8_t channelIndex { 0 };
};

codecStreamBridge_.submitInput(localBuffer, currentSampleRate_, channelIndex);

if (codecStreamBridge_.popEncoded(frame))
    framedTransport_.enqueueOutbound(frame.payload, frame.channelIndex);
```
Source: local code inspection of `src/audio/CodecStreamBridge.h`, `src/audio/CodecStreamBridge.cpp`, and `src/net/FramedSocketTransport.cpp`

### Pattern 4: Fixed-Cap Remote Route Map Replaces `HostRoutingProofRoute`
**What:** Replace the one-source proof route seam with a small fixed-cap `sourceId -> outputBusIndex` assignment map where `0` means main output.
**When to use:** Remote routing implementation and state restore.
**Why:** This preserves the host-owned fixed-bus model while allowing more than one routed remote return without committing to dynamic bus counts or a patchbay.
**Example:**
```cpp
// 0 = FLJ Main Output, 1..N = fixed extra routed output buses.
std::unordered_map<std::string, int> remoteOutputAssignments_;

const int busIndex = lookupAssignedOutput(sourceId);
auto& target = busIndex == 0 ? mainOutput : routedOutputs[busIndex - 1];
```
Source: local code inspection of `src/plugin/FamaLamaJamAudioProcessor.h` and `src/plugin/FamaLamaJamAudioProcessor.cpp`

### Pattern 5: Minimal Editor via Local Header + Remote-Strip Dropdown
**What:** Put global local-send controls and `Add channel` in a local-group header, and put a narrow output dropdown only on remote strips.
**When to use:** Current editor, before Phase 9.
**Why:** The current mixer rebuild logic already keeps widget order stable and avoids full widget churn on unchanged strips. A header plus dropdown fits that model without forcing a layout rewrite.
**Example:**
```cpp
// Local group header:
// [Local Sends] [Transmit button] [Voice toggle] [Add channel]
//
// Remote strip:
// title | subtitle
// [meter] [gain] [pan] [solo] [mute] [Output dropdown]
```
Source: local code inspection of `src/plugin/FamaLamaJamAudioProcessorEditor.cpp`

### Recommended Evolution Of `HostRoutingProof`
**What:** Keep the proof names and bus indices, but retire `HostRoutingProof` as the main implementation model.
**When to use:** At the start of implementation.
**Why:** `HostRoutingProof` is still useful as a narrow diagnostic/test seam, but it is not rich enough for:
- two local transmit slots,
- hidden/restored extra local channels,
- persisted remote route assignments,
- more than one routed remote bus,
- or channel-indexed outbound uploads.

**Recommended replacement shape:**
```cpp
struct FixedRoutingConfig
{
    std::array<LocalInputSlot, 2> localInputs;
    int extraRemoteOutputPairs { 2 }; // lock in Wave 0; do not make dynamic
};

struct FixedRoutingRuntimeState
{
    LocalSendGlobalState localGlobal;
    std::unordered_map<std::string, LocalChannelState> localChannelsById;
    std::unordered_map<std::string, int> remoteOutputAssignments;
};
```

### Anti-Patterns to Avoid
- **Modeling extra local slots as more `LocalMonitor` clones:** restore currently forces `LocalMonitor` strips visible and active again.
- **Using display names as persistence keys:** rename must not destroy mix state or route assignments.
- **Mirroring routed remotes to both main and aux:** the locked decision is explicit that reassigned remotes leave main completely.
- **Adding dynamic bus APIs now:** do not override `canAddBus` or design a runtime bus manager for this phase.
- **Putting global `Transmit` and `Voice` back onto every strip:** that directly reopens a solved UX decision.
- **Trying to infer the sidechain source track name from JUCE host track callbacks:** `updateTrackProperties()` tells the plugin about its host track, not the separate Ableton track routed into `Local Send 2` (this is an inference from the JUCE API docs).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bus-to-buffer mapping | Custom absolute-channel math everywhere | `getBusBuffer()` and existing JUCE bus layout APIs | Official JUCE guidance already covers multi-bus access and buffer ownership |
| Multi-local sender transport | A second special-case upload path for `Local Send 2` | Extend `CodecStreamBridge` and `FramedSocketTransport` metadata | Keeps all local channels on one consistent NINJAM send path |
| Local strip identity | IDs derived from editable names | Stable slot IDs such as `local-main` and `local-send-2` | Rename must not break restore or hide/show behavior |
| DAW routing UX | Plugin-owned Ableton track browser | Host routing through Live's existing routing UI | Research already approved the host-owned workflow |
| Remote routing semantics | Route-and-mirror logic | One selected output target per remote source | Simpler, matches the locked UX, and keeps isolation testable |

**Key insight:** The deceptively hard part is not the extra JUCE bus. It is getting the local upload identity, persistence, and UI model aligned so "second local channel" means a real second NINJAM channel all the way out to the protocol.

## Common Pitfalls

### Pitfall 1: The UI Shows Two Local Strips But The Server Still Sees One Channel
**What goes wrong:** `Main` and `Local Send 2` meter separately, but the remote room still receives only channel `0`.
**Why it happens:** The current outbound path has one local channel metadata slot and `writeUploadInterval()` hardcodes `beginPayload[24] = 0`.
**How to avoid:** Carry channel index and channel info through `CodecStreamBridge` and `FramedSocketTransport`, then extend the test harness to assert it.
**Warning signs:** `Local Send 2` looks alive locally, but another client never sees a second channel.

### Pitfall 2: Hidden Extra Local Channels Reappear After Restore
**What goes wrong:** Removing the extra local strip only hides it during the session; reopening the project revives it visibly.
**Why it happens:** Restore currently forces any `LocalMonitor` strip to `active=true` and `visible=true`.
**How to avoid:** Persist local slot visibility explicitly and stop treating all local strips as the special singleton kind.
**Warning signs:** The state blob restores mix values correctly but the extra strip always comes back visible.

### Pitfall 3: Rename Breaks Mix Persistence
**What goes wrong:** Renaming `Local Send 2` to `Bass` causes its stored gain/pan/mute state to reset or move.
**Why it happens:** Persistence was tied to the user-visible label instead of a stable slot ID.
**How to avoid:** Keep immutable slot IDs and persist editable labels separately.
**Warning signs:** A rename changes the `sourceId` used in state serialization.

### Pitfall 4: Output Assignments Offer Buses The Host Has Disabled
**What goes wrong:** A remote strip can be assigned to a bus that is currently unavailable in the active layout, resulting in silent output.
**Why it happens:** VST3 buses are host-activated; default-active is only a hint.
**How to avoid:** Build the dropdown from the current enabled bus layout or fall back invalid restored assignments to `FLJ Main Output`.
**Warning signs:** The dropdown lists an aux bus, but the host routing chooser does not.

### Pitfall 5: Minimal UI Slips Into Phase 9 Layout Work
**What goes wrong:** The remote output dropdown or local channel controls force a broader strip redesign.
**Why it happens:** Global controls and per-strip controls are not separated early enough.
**How to avoid:** Put global controls in the local-group header, keep remote routing remote-only, and respect the current vertical strip geometry.
**Warning signs:** The work starts changing strip density, horizontal flow, or meter/control arrangement beyond what the dropdown/header needs.

### Pitfall 6: Expecting Ableton To Supply The Sidechain Source Track Name
**What goes wrong:** The extra local slot gets a misleading or empty default label because the implementation waits for source-track metadata that never arrives.
**Why it happens:** JUCE exposes the processor host track properties, but that is not the same thing as the separate track feeding the sidechain bus.
**How to avoid:** Default from the bus name and let user rename it immediately.
**Warning signs:** The phase plan depends on source-track-name discovery before the second local strip can be named.

## Open Questions

1. **How many fixed extra remote output pairs should this phase ship?**
   - What we know: the proof phase validated one extra routed pair, and the locked UI wording anticipates more than one.
   - What's unclear: the exact cap is not yet locked for `08.3.4.1`.
   - Recommendation: lock a compile-time constant in Wave 0. `2` extra pairs is the largest sensible cap for this phase; more drifts toward patchbay work.

2. **Should hidden extra local channels persist across project reopen or only within the running session?**
   - What we know: the discretion section allows either session-only or project-persistent behavior as long as the user-visible behavior is "hide and remember."
   - What's unclear: the user did not explicitly require cross-project restore.
   - Recommendation: persist it in project state. The project already persists mixer strip state, and project-persistent restore is low incremental cost once local slots use stable IDs.

3. **Where should global `Transmit` and `Voice` live in the current UI?**
   - What we know: the locked decision says they are global and should not consume per-strip space.
   - What's unclear: exact placement.
   - Recommendation: keep them in the local-group header next to `Add channel`, not in the transport row and not repeated per strip.
