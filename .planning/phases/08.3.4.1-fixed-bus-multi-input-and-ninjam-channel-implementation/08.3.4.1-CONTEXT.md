# Phase 08.3.4.1: Fixed-Bus Multi-Input And NINJAM Channel Implementation - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement the first practical multi-I/O workflow validated by Phase 08.3.4:
- two local transmit paths via `Main` and `Local Send 2`
- fixed extra FLJ output pairs for routed remote returns using host-owned Ableton routing

This phase is about the first real implementation of bus-based routing, not broader dynamic bus counts, not plugin-owned Ableton track enumeration, and not the larger Phase 9 mixer/layout redesign.

</domain>

<decisions>
## Implementation Decisions

### Local strip model
- Extra local transmit channels should appear as full local strips, not compact subrows.
- Extra local transmit channels are user-created; they should not always exist by default.
- The user should create them through an inline `Add channel` control near the local-strip area.
- The default name should come from the host or bus name when available, otherwise `Local Send 2`.
- Local channel names are important and must be user-editable for meaningful instrument labels like `Bass` or `Guitar`.

### Per-channel controls
- `Transmit` becomes a global local-send control rather than a per-local-channel control.
- `Voice` becomes a global local-send mode rather than a per-local-channel control.
- When voice mode is enabled, all active local channels transmit voice.
- Each local channel still keeps its own gain, pan, mute, and solo controls.
- Removing an extra local channel should hide it but preserve its last settings so it can be restored easily within the session.

### Remote output routing UX
- Remote output assignment should live directly on each remote strip as a small dropdown.
- The dropdown should show explicit host-facing labels:
  - `FLJ Main Output`
  - `Remote Out 1`
  - `Remote Out 2`
  - and so on as supported
- If a remote channel is routed away from the main output, it should be removed from the FLJ main output completely rather than mirrored.

### Claude's Discretion
- Exact wording and visual style of the `Add channel` control.
- Exact inline editing affordance for local channel renaming.
- Exact placement and width of the remote output dropdown within the current pre-Phase-9 strip layout.
- Exact session-only versus project-persistent handling for hidden extra local channels, as long as the user-visible behavior matches “hide and remember”.

</decisions>

<specifics>
## Specific Ideas

- The second local transmit channel should behave like a real additional NINJAM channel, not just a hidden sidechain path.
- The user wants meaningful local channel names because upstream clients like JamTaba and ReaNINJAM allow channel names such as `Bass` or `Guitar`.
- The routing model should stay host-owned:
  - FLJ exposes buses
  - Ableton routes tracks to those buses
  - FLJ does not try to list all Ableton tracks internally
- Keeping global `Transmit` and `Voice` controls helps free space in the strip layout before the larger mixer redesign.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `FamaLamaJamAudioProcessor` already has fixed proof buses declared for `Local Send 2` and `Remote Out 1`.
- `HostRoutingProof` and `HostRoutingProofRoute` already exist as processor-owned seams for the first bus-based implementation.
- The current mixer already supports strip-level gain, pan, mute, and solo state.
- The editor already has local transmit and voice toggle handlers that can be repositioned or made global without inventing a separate transport subsystem.

### Established Patterns
- Voice and transmit state are currently surfaced from processor-owned snapshot state rather than editor-local business logic.
- Remote routing is already treated post-decode in the proof build, which should remain true in the implementation phase.
- The project prefers explicit user-facing status and narrow, testable seams before broader UI redesign work.

### Integration Points
- Bus declaration and per-bus processing live in:
  - `src/plugin/FamaLamaJamAudioProcessor.h`
  - `src/plugin/FamaLamaJamAudioProcessor.cpp`
- Strip rendering and inline controls live in:
  - `src/plugin/FamaLamaJamAudioProcessorEditor.h`
  - `src/plugin/FamaLamaJamAudioProcessorEditor.cpp`
- Current proof routing and bus names already appear in:
  - `HostRoutingProof`
  - `kHostRoutingProofAuxInputBusName`
  - `kHostRoutingProofAuxOutputBusName`

</code_context>

<deferred>
## Deferred Ideas

- Arbitrary dynamic add/remove bus counts beyond the first `Main + Local Send 2` implementation remain a later expansion.
- Broader remote output fanout beyond the first fixed extra pairs remains later expansion work.
- Full mixer-density redesign, horizontal strips, embedded volume-in-meter controls, and broader layout changes remain Phase 9 work.
- Plugin-owned Ableton track enumeration remains out of scope; host-owned routing is the decided model.

</deferred>
