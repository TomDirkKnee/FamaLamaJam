## Post-Checkpoint Regression Review

After the original blocked checkpoint, the fixed-bus implementation moved forward through several narrow fixes:
- real packed multi-channel `ClientSetChannelInfo`
- hidden extra-channel unpublish
- local `Local Send 2` metering
- local `Local Send 2` foldback into FLJ main output
- snapshot-based local upload capture from copied host input buffers

Those changes got the second local send publishing and made remote `jim#1` decode with non-zero RMS in live Ableton tests. However, a later troubleshooting change introduced a likely receive-scheduling regression:
- processor-side live remote playback began consuming inferred boundary-generation metadata from download-begin events
- sidechain audio became audible remotely, but dropped out roughly once per interval
- in some runs the main remote channel began showing similar queued or refresh churn

That regression review is documented here:
- `.planning/debug/2026-04-06-fixed-bus-receive-timing-regression.md`

The processor-side live boundary inference has now been rolled back while preserving the working fixed-bus capture changes. Focused automation after the rollback is green again:
- `famalamajam_tests.exe "[plugin_host_multi_io_routing],[plugin_remote_interval_alignment],[plugin_voice_mode_transport]"`
- result: pass, `179` assertions across `18` test cases

The rollback build cleared the real Jim/Ben sidechain audio path. The plan now needs either:
- an explicit reopen persistence rerun to close `P8.3.4.1-ROUTE-06`, or
- a deliberate decision to defer reopen persistence and removable-channel UX to the next phase.
